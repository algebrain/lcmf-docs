# Документация: App-level слой frontend

Этот документ описывает app-level слой в `LCMF`.

Речь здесь идет о той части frontend-приложения, которая собирает множество
маленьких модулей в одну систему и передает им общую инфраструктуру.

Если вам нужна общая архитектурная рамка, сначала прочитайте
[`ARCH.md`](./ARCH.md).
Если вам нужна форма отдельного модуля, сначала прочитайте
[`MODULE.md`](./MODULE.md).
Если вам нужен механизм публичных синхронных чтений, переходите в
[`REGISTRY.md`](./REGISTRY.md).

## 1. Что такое app-level слой в `LCMF`

App-level слой в `LCMF` — это слой сборки приложения.

Он не владеет предметной логикой модулей.
Его задача:

- создать общую шину;
- создать общий `registry` публичных чтений;
- создать общий HTTP runtime;
- создать realtime runtime, если он нужен;
- подготовить прочие общие зависимости, если они действительно нужны проекту;
- вызвать `init!` модулей;
- выполнить стартовые проверки;
- собрать приложение из независимых маленьких модулей.

Коротко:

- app owns infrastructure
- modules own state and behavior

Главный practically useful смысл app-level слоя такой:

- маленькие модули не должны каждый раз собирать свою инфраструктуру с нуля;
- app-level слой делает систему из многих модулей дешевой в подключении и
  сопровождении.

## 2. Что именно собирает приложение

Обычная картина app-level слоя такая:

- `bus`
- `registry`
- HTTP runtime
- realtime runtime, если он нужен
- конфиг
- опциональные технические инструменты вроде `logger`, если проект вообще
  хочет их использовать
- app-level связку между этими частями

Важно:

- приложение не должно забирать владение состоянием у модулей;
- приложение не должно становиться скрытым центром предметной координации;
- приложение не должно вынуждать каждый маленький модуль тащить лишний
  служебный код.

## 3. App-level слой и состояние

App-level слой может собирать модульные участки состояния в runtime
приложения.

Но это не должно читаться как:

- "настоящий центр системы — это один большой app-state";
- "модули — это просто поля общего store";
- "приложение знает внутреннюю форму всей системы и управляет ей напрямую".

Смысл app-level слоя другой:

- приложение собирает окружение, внутри которого каждый модуль владеет своим
  участком;
- приложение не получает права читать и менять внутренности модулей как ему
  удобно;
- общая физическая форма хранения состояния вторична по сравнению с владением и
  границами модулей.

На этом этапе `LCMF` не обязан фиксировать одну физическую форму runtime
состояния.
Возможны разные реализации, пока сохраняются базовые инварианты:

- владение состоянием остается у модуля;
- межмодульная координация идет через `bus`;
- публичные синхронные чтения идут через `registry`;
- внутреннее состояние другого модуля не читается напрямую.

### Пример возможной формы app-level состояния

```clojure
(defn make-app-state []
  {:booking (atom booking.state/initial-state)
   :accounts (atom accounts.state/initial-state)
   :catalog (atom catalog.state/initial-state)})
```

Это только пример физической сборки.

Важно не то, что это карта `atom`-ов, а то, что:

- модули не теряют владение;
- app-level слой не превращает эту форму в новый канал межмодульной
  координации.

## 4. Что модуль получает от app-level слоя

`LCMF`-модуль получает от приложения общие зависимости, но не должен получать
право свободно обходить архитектурные границы.

Практически это означает:

- модуль получает `bus`;
- модуль получает `registry`;
- модуль получает HTTP runtime, если он ему нужен;
- модуль может получать доступ к своему состоянию или к выделенному runtime
  своего состояния;
- модуль может получать опциональные технические инструменты, если проект их
  использует;
- модуль не должен использовать app-level связку как способ читать чужое
  внутреннее состояние.

Главное правило здесь простое:

- модуль должен получать только те зависимости, которые ему реально нужны;
- app-level слой не должен передавать модулю лишние общие объекты "на всякий
  случай".

## 5. Допустимая и недопустимая форма зависимостей

### Пример допустимой формы

```clojure
(defn init-app! []
  (let [bus (bus/make-bus)
        registry (registry/make-registry)
        app-state (make-app-state)
        http-runtime (http/make-runtime)]
    (booking/init! {:bus bus
                    :registry registry
                    :http http-runtime
                    :state (:booking app-state)})
    (accounts/init! {:bus bus
                     :registry registry
                     :http http-runtime
                     :state (:accounts app-state)})))
```

Что здесь важно:

- приложение передает модулю его зависимости;
- модуль получает только свой участок состояния;
- модуль не получает общий обзор внутренних данных соседей.

### Пример недопустимой формы

```clojure
;; Плохо: модуль получает весь app-state и может читать чужие внутренности.
(booking/init! {:bus bus
                :registry registry
                :http http-runtime
                :app-state app-state})
```

Но проблема не только в передаче всего `app-state`.

Плохой будет любая app-level связка, в которой app-level слой:

- передает модулю лишние данные о соседях;
- создает неявный обход контрактов;
- превращается в скрытый центр предметной координации.

## 6. App-level слой и шина

Шина создается на уровне приложения и передается в модули как общая
инфраструктурная зависимость.

Это важно по двум причинам:

- все модули координируются через один и тот же `bus`;
- шина остается app-level механизмом, а не внутренней деталью отдельного
  модуля.

Простой пример:

```clojure
(defn make-system []
  {:bus (bus/make-bus)})
```

App-level слой не должен подменять шину собственной предметной логикой.
Он должен только дать модулям общий канал координации.

## 7. App-level слой и HTTP runtime

HTTP runtime тоже создается на уровне приложения.

Это нужно для того, чтобы маленькие модули не дублировали:

- одинаковый способ выполнения запросов;
- одинаковую нормализацию ошибок;
- одинаковую обработку auth-сценариев;
- одинаковую передачу технических данных трассировки.

Модуль при этом по-прежнему владеет своими операциями поверх этого общего
уровня.

Простой пример:

```clojure
(defn make-system []
  {:http (http/make-runtime {:base-url "/api"})})
```

## 8. App-level слой и стартовая последовательность

Обычный порядок app-level сборки такой:

- создать `bus`;
- создать `registry`;
- создать HTTP runtime;
- создать realtime runtime, если он нужен;
- подготовить прочие общие зависимости, если проект их использует;
- вызвать `init!` модулей;
- выполнить стартовые проверки;
- собрать финальное приложение.

### Пример app-level сборки

```clojure
(defn init-app! []
  (let [bus (bus/make-bus)
        registry (registry/make-registry)
        app-state (make-app-state)
        http-runtime (http/make-runtime {:base-url "/api"})]
    (accounts/init! {:bus bus
                     :registry registry
                     :http http-runtime
                     :state (:accounts app-state)})
    (catalog/init! {:bus bus
                    :registry registry
                    :http http-runtime
                    :state (:catalog app-state)})
    (booking/init! {:bus bus
                    :registry registry
                    :http http-runtime
                    :state (:booking app-state)})
    (registry/assert-requirements! registry)
    {:bus bus
     :registry registry
     :state app-state
     :http http-runtime}))
```

Но здесь важно понимать еще одну practically useful вещь:

- app-level слой не должен превращаться в узкое ручное горлышко;
- подключение нового маленького модуля должно оставаться дешевым;
- если для добавления модуля нужно слишком много ручной связки, это сигнал
  улучшать поддерживающие библиотеки и соглашения.

## 9. Что app-level слой не должен делать

App-level слой не должен:

- забирать владение состоянием у модулей;
- превращать app-state в общий обход границ модулей;
- подменять `bus` прямыми вызовами между модулями;
- подменять `registry` прямым чтением чужих внутренних данных;
- втягивать в `LCMF` устройство компонентов и view-layer;
- накапливать предметную координацию между модулями у себя;
- становиться местом, где приложение "договаривается за модули" вместо их
  явных контрактов.

Если в app-level коде начинает расти предметная логика, это архитектурный
сигнал вернуться к границе:

- app собирает;
- модуль ведет себя сам.

## 10. Минимальный итог

App-level слой в `LCMF` — это слой сборки, а не слой предметной логики.

Он нужен не сам по себе.
Он нужен для того, чтобы система из многих маленьких модулей оставалась:

- дешевой в подключении;
- предсказуемой в сборке;
- дисциплинированной по границам;
- свободной от лишнего дублирования инфраструктуры.

Приложение создает общую инфраструктуру, а модули через нее подключаются к
системе и дальше координируются прежде всего через `bus`.
