# Документация: Frontend-модули

Этот документ описывает стандартный подход к созданию маленьких модулей в
`LCMF`.

Frontend-модуль в этой архитектуре — это маленькая единица состояния со своей
границей ответственности, которая участвует в общей предметной картине
frontend прежде всего через:

- сообщения шины;
- публичные чтения через `registry`;
- свои операции, связанные с backend.

Если вам нужна не module-level форма, а app-level сборка вокруг `init!`,
сначала прочитайте [`APP.md`](./APP.md).
Если нужен типовой helper для состояния одного запроса внутри модуля,
смотрите также [`REQUEST_STATE.md`](./REQUEST_STATE.md).

## 1. Главный ориентир

Хороший модуль в `LCMF` должен оставаться маленьким.

Это означает:

- у модуля узкая граница ответственности;
- у модуля небольшой участок состояния;
- публичную поверхность модуля можно коротко описать;
- модуль не требует тяжелого каркаса, если его задача проста.

Важно:

- модуль не должен выглядеть как "маленькое приложение внутри приложения";
- модуль не обязан использовать все возможные механизмы сразу;
- если модулю не нужны подписки, HTTP или внешние чтения, он не обязан
  имитировать их ради формы.

## 2. Анатомия модуля

Каждый модуль обычно предоставляет функцию инициализации `init!`, которая
служит его точкой входа.

Сигнатура:

```clojure
(defn init! [deps])
```

Параметр `deps` — карта зависимостей.

Обычные ключи:

- `:bus` — экземпляр frontend message bus
- `:registry` — реестр публичных чтений
- `:http` — app-level HTTP runtime, если модулю действительно нужен HTTP
- `:logger` — опциональная функция для технических сообщений, если проект
  вообще использует такой механизм
- `:config` — уже загруженный конфиг приложения или модуля, если он нужен

Важно:

- `init!` является стандартной формой;
- но маленький модуль может использовать только ту часть зависимостей, которая
  ему реально нужна;
- отсутствие HTTP, подписок или `logger` не делает модуль "неправильным".

Внутри `init!` модуль может:

- зарегистрировать свои публичные чтения;
- объявить обязательные внешние чтения;
- объявить обязательные app-level зависимости, если проект использует явный
  dependency contract;
- подписаться на нужные сообщения;
- подготовить свои операции;
- вернуть наружу свои модульные дескрипторы, если это действительно нужно
  приложению;
- вернуть cleanup hook или другой lifecycle descriptor, если это реально нужно
  app-level сборке.

## 2.1 `init!` как часть запуска приложения

Хотя `init!` является точкой входа модуля, он не существует сам по себе.

Для `LCMF` минимально полезная картина такая:

- `init!` вызывается как часть общего запуска приложения;
- ошибка wiring или исключение внутри `init!` считаются ошибкой сборки
  приложения;
- модуль может рассчитывать только на те зависимости, которые действительно
  входят в его контракт с приложением;
- если проекту это нужно, `init!` может вернуть средство очистки или
  завершения, которое приложение использует при откате неудачного запуска или
  при завершении работы.

Важно не сделать отсюда лишний вывод.

Это не означает, что:

- средство очистки или завершения становится главным API модуля для соседних
  модулей;
- механизм жизненного цикла заменяет `bus` или `registry`;
- app-level слой получает право управлять предметной логикой модуля.

Смысл только в одном:

кроме межмодульной предметной поверхности у модуля может существовать отдельная
поверхность для его встраивания в приложение.

## 3. Минимальный пример маленького модуля

Это пример маленького `LCMF`-модуля без HTTP и без внешних зависимостей.

```clojure
(ns booking.filter
  (:require [lcmf.bus :as bus]
            [lcmf.registry :as registry]))

(def module-id :booking-filter)

(def initial-state
  {:status :all})

(defn- make-state []
  (atom initial-state))

(defn- current-filter [state]
  {:status (:status @state)})

(defn- register-public-reads! [registry-instance state]
  (registry/register-provider!
   registry-instance
   {:provider-id :booking-filter/current
    :module module-id
    :provider-fn (fn [_]
                   (current-filter state))}))

(defn init!
  [{:keys [registry]}]
  (let [state (make-state)]
    (register-public-reads! registry state)
    {:module module-id
     :state state}))
```

Этот пример важен не полнотой, а легкостью:

- модуль маленький;
- модуль не обязан иметь HTTP;
- модуль не обязан иметь подписки;
- модуль не обязан иметь тяжелую внешнюю поверхность.

## 4. Работа с шиной сообщений

Шина является главным способом межмодульной координации в `LCMF`.

Модуль может:

- публиковать сообщения о произошедших фактах;
- подписываться на сообщения других модулей;
- реагировать на app-owned сообщения, если это часть его зависимости.

При этом модуль не должен знать, кто именно подписан на его сообщения.

### Пример публикации сообщения

```clojure
(defn- publish-booking-created! [bus-instance booking]
  (bus/publish! bus-instance
                :booking/created
                {:id (:id booking)
                 :slot-id (:slot-id booking)
                 :user-id (:user-id booking)}
                {:module module-id}))
```

### Пример реакции на сообщение другого модуля

```clojure
(defn- handle-account-signed-out [state _bus envelope]
  (let [user-id (get-in envelope [:payload :user-id])]
    (swap! state update :items
           (fn [items]
             (into {}
                   (remove (fn [[_ booking]]
                             (= user-id (:user-id booking))))
                   items)))))

(defn- subscribe! [bus-instance state]
  (bus/subscribe! bus-instance
                  :accounts/signed-out
                  (fn [bus envelope]
                    (handle-account-signed-out state bus envelope))
                  {:module module-id}))
```

Если обработчик сообщения публикует новое сообщение, он должен сохранять
причинно-следственную связность через `:parent-envelope`, если это
предусмотрено API конкретной библиотеки шины.

## 5. Чем модуль владеет

Модуль в `LCMF` владеет:

- своим участком состояния;
- своими переходами состояния;
- своими сообщениями;
- своими публичными чтениями;
- своими предметными операциями;
- тем, как результат этих операций отражается в его состоянии.

Это не означает, что модуль владеет всей инфраструктурой вокруг себя.

Приложение по-прежнему владеет:

- общей шиной;
- общим сетевым runtime;
- общим `registry`;
- общими правилами обработки ошибок и трассировки;
- общими техническими инструментами вроде `logger`, если они вообще нужны.

То есть модуль владеет своим поведением, но опирается на общую app-level
инфраструктуру.

## 6. Что модуль экспортирует для других модулей

Для текущей рамки `LCMF` важно жестко различать два направления публичности:

- в сторону других модулей;
- в сторону интерфейса.

Этот документ сосредоточен прежде всего на первом направлении.

Для других модулей нормальными публичными поверхностями являются:

- сообщения, которые модуль публикует в шину;
- публичные чтения через `registry`.

Это и есть основная межмодульная предметная поверхность.

## 7. Что модуль не должен экспортировать как межмодульный API

Для соседних модулей модуль не должен становиться набором вызываемых функций.

Это важно сформулировать прямо:

- прямой вызов внутренних функций чужого модуля запрещен;
- знание внутренней формы его состояния запрещено;
- главный межмодульный API — это не "набор функций модуля", а его сообщения и
  публичные чтения.

У модуля может существовать отдельная поверхность в сторону интерфейса.
Но это другой вектор публичности и он не является главным предметом этого
документа.

## 8. Публичные синхронные чтения

Публичное чтение должно давать достаточно данных для межмодульной координации,
но не превращаться в главный способ архитектурной координации.

В `LCMF` эту роль выполняет шина.

### Пример публичного чтения

```clojure
(defn- get-booking-by-id [state booking-id]
  (get-in @state [:items booking-id]))

(defn- register-public-reads! [registry-instance state]
  (registry/register-provider!
   registry-instance
   {:provider-id :booking/get-by-id
    :module module-id
    :provider-fn (fn [{:keys [booking-id]}]
                   (when booking-id
                     (get-booking-by-id state booking-id)))})
  (registry/register-provider!
   registry-instance
   {:provider-id :booking/list-bookings
    :module module-id
    :provider-fn (fn [_]
                   (vals (:items @state)))}))
```

Главное правило:

- provider описывает внешний сценарий чтения;
- provider не раскрывает всю внутреннюю форму состояния.

## 9. Что остается внутренним

Внутренними для модуля остаются:

- точная форма его участка состояния;
- внутренние функции переходов;
- служебные поля request lifecycle;
- промежуточные вычисления;
- внутренняя организация файла или файлов модуля.

Другой модуль не должен читать эти данные напрямую и не должен строить свою
логику на знании внутренней структуры чужого состояния.

## 10. Работа с HTTP

HTTP-операции являются частью модульного поведения, но не делают модуль
владельцем всего сетевого слоя.

Модуль сам определяет:

- какие запросы ему нужны;
- как результат отражается в его состоянии;
- какие сообщения публикуются по итогам;
- как обрабатываются предметные успех и ошибка.

Приложение же дает ему общий HTTP runtime.

### Пример HTTP-операции модуля

```clojure
(defn create-booking!
  [{:keys [state http bus]} payload]
  (mark-create-pending! state)
  (-> (http/request! http
                     {:method :post
                      :path "/bookings"
                      :json payload})
      (.then (fn [booking]
               (store-created-booking! state booking)
               (publish-booking-created! bus booking)
               booking))
      (.catch (fn [error]
                (swap! state assoc
                       :create-request {:status :error
                                        :error error})
                (throw error)))))
```

Важно:

- такая операция является частью поведения модуля;
- но для других модулей она не должна считаться главным межмодульным API;
- межмодульная картина по-прежнему должна читаться через `bus` и `registry`.

## 11. Обязательные межмодульные чтения

Если модулю нужны публичные данные другого модуля, он должен явно объявить эту
зависимость через `registry`.

Это нужно для того, чтобы приложение могло проверить все обязательные
межмодульные чтения на старте.

### Пример объявления зависимостей

```clojure
(defn- declare-requirements! [registry-instance]
  (registry/declare-requirements! registry-instance
                                  module-id
                                  #{:accounts/get-by-id
                                    :catalog/get-slot-by-id}))
```

Само приложение после инициализации модулей должно выполнить общую проверку
обязательных зависимостей.

Важно:

- `registry` не заменяет `bus`;
- если через публичные чтения начинает проходить основная координация между
  модулями, это сигнал пересмотреть границы модулей.

## 12. Наблюдаемость и технические сообщения

Для малого проекта frontend-модуль не обязан иметь тяжелую техническую обвязку.

Если проект использует `logger`, модуль может писать короткие структурированные
технические сообщения через переданную функцию.

Но здесь важен правильный акцент:

- `logger` не является главной несущей осью модуля;
- отсутствие logger не делает модуль архитектурно неправильным;
- модуль не должен строить собственную отдельную систему журналирования.

### Пример технического сообщения

```clojure
(when logger
  (logger :info {:component ::booking
                 :event :booking/create-started
                 :slot-id (:slot-id payload)
                 :user-id (:user-id payload)}))
```

## 13. Минимальный итог

Хороший frontend-модуль в `LCMF` — это маленькая единица состояния, которая:

- владеет своим участком состояния;
- остается дешевой по форме и не тянет лишний каркас;
- координируется с другими модулями прежде всего через шину;
- читает чужие данные только через публичный контракт;
- использует общий app-level сетевой слой;
- не превращает свои верхнеуровневые функции в основной межмодульный API.
