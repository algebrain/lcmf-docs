# Документация: Трассировка и минимальный технический след

Этот документ описывает минимальный уровень трассировки и технического следа,
который нужен `LCMF` для совместимости с backend, построенным по принципам
[`LCMM`](https://github.com/algebrain/lcmm-docs).

Речь идет о минимуме, который реально нужен на старте:

- сохранить `correlation-id` и `request-id`;
- не терять причинно-следственную связность внутри `bus`;
- иметь достаточно данных для bug report и support/debug payload;
- не логировать и не передавать лишние секреты.

Отдельная большая frontend-подсистема наблюдаемости здесь не вводится.

Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для HTTP-части см. [`HTTP.md`](./HTTP.md).
Для шины см. [`BUS.md`](./BUS.md).

Важно держать этот текст в согласии с backend-ориентирами:

- [`TRACE_FLOW.md`](https://github.com/algebrain/lcmm-docs/blob/main/docs/TRACE_FLOW.md)
- [`LOGGING.md`](https://github.com/algebrain/lcmm-docs/blob/main/docs/LOGGING.md)
- [`OBSERVABILITY.md`](https://github.com/algebrain/lcmm-docs/blob/main/docs/OBSERVABILITY.md)

Но и весь вес backend-наблюдаемости этот frontend-документ на клиент не
переносит.

## 1. Что здесь считается достаточным минимумом

Для `LCMF` на первом этапе достаточно следующего:

- HTTP runtime умеет читать `correlation-id` и `request-id` из ответа и ошибки;
- `bus` умеет переносить `correlation-id` и `parent-envelope` по цепочке;
- приложение умеет при ошибке собрать короткий support/debug payload без
  секретов;
- проект может при необходимости оставить короткий технический след, но не
  обязан строить для этого отдельный тяжелый logging-слой.

Этого достаточно, чтобы frontend можно было связать с backend-трассой и не
терять ориентиры при разборе проблем.

## 2. `correlation-id` и `request-id`

Для минимальной совместимости с
[`LCMM`](https://github.com/algebrain/lcmm-docs) frontend должен различать два
идентификатора:

- `correlation-id` — идентификатор общей цепочки;
- `request-id` — идентификатор конкретного HTTP-запроса.

Практический смысл:

- `correlation-id` помогает backend-команде найти всю цепочку;
- `request-id` помогает найти конкретный запрос;
- если доступны оба значения, сохранять нужно оба.

### Пример приведенной HTTP-ошибки

Это пример формы ошибки на стороне frontend runtime.

```clojure
{:status 503
 :code "dependency_unavailable"
 :message "Dependency unavailable"
 :details nil
 :correlation-id "c1a2..."
 :request-id "r9f0..."
 :retry-after-sec 5}
```

## 3. Причинно-следственная связность в `bus`

Если `LCMF`-модуль публикует сообщение в ответ на уже полученное сообщение, он
должен продолжать цепочку, а не начинать ее заново.

Практический минимум здесь такой:

- конверт шины должен содержать `correlation-id`;
- следующее сообщение должно получать `parent-envelope`;
- этого достаточно, чтобы не рвать frontend-цепочку.

Речь идет только о дисциплине продолжения уже существующего контекста.
Отдельная сложная trace-модель здесь не вводится.

### Пример публикации следующего сообщения

Это пример внутреннего обработчика `LCMF`-модуля.

```clojure
(bus/subscribe! bus
                :booking/created
                (fn [bus envelope]
                  (bus/publish! bus
                                :notify/booking-created
                                {:booking-id (get-in envelope [:payload :id])}
                                {:parent-envelope envelope
                                 :module :notify})))
```

## 4. Короткий технический след

Для раннего `LCMF` полезно различать две вещи:

- короткий технический след, который помогает понять, что произошло;
- и отдельную большую подсистему клиентской наблюдаемости.

В этом документе нужна только первая вещь.

Проект может реализовать ее по-разному:

- через очень простой `logger`;
- через безопасный app-level helper;
- через интеграцию с внешним error-reporting инструментом;
- или вообще только через сбор payload в момент ошибки.

Правильная мысль здесь такая:

- минимальный технический след полезен;
- но его форма остается решением проекта;
- `LCMF` не требует раннего frontend-аналога backend logging stack.

Именно поэтому не стоит считать заранее, что отдельный `logger` уже является
обязательной частью `LCMF`.

## 5. Support/debug payload

Если пользователю или разработчику нужно передать информацию о проблеме,
frontend должен уметь собрать короткий support/debug payload.

Минимально полезный набор:

- имя сценария или экрана;
- HTTP method и endpoint, если ошибка была связана с HTTP;
- `status` и `code`;
- `correlation-id`;
- `request-id`, если он доступен;
- startup phase или startup status, если ошибка произошла во время app-level
  сборки;
- время ошибки.

### Пример support/debug payload

Это пример app-level payload для поддержки или bug report.

```clojure
{:screen :booking-page
 :operation :create-booking
 :method :post
 :path "/bookings"
 :status 409
 :code "slot_not_open"
 :correlation-id "c1a2..."
 :request-id "r9f0..."
 :timestamp "2026-04-04T12:00:00Z"}
```

Полезно жестко различать:

- постоянный минимальный технический след;
- support/debug payload, который собирается по ситуации.

Это помогает не тащить в клиент лишнее постоянное логирование только ради того,
чтобы однажды было что приложить к bug report.

## 6. Что нельзя включать в технический след и support/debug payload

В технические данные клиента нельзя включать:

- access token;
- cookie;
- session id;
- секреты из заголовков;
- сырой payload целиком, если в нем есть чувствительные данные.

Практическое правило:

- frontend должен помогать отладке;
- frontend не должен ради отладки нарушать безопасное поведение клиента.

## 7. Что здесь не нужно усложнять

На первом этапе `LCMF` не обязан:

- вводить отдельную систему метрик уровня Prometheus;
- требовать отдельный frontend observability registry;
- покрывать каждый экран и каждую функцию телеметрией;
- делать сложную span-модель внутри клиента;
- строить frontend-аналог [`lcmm-observe`](https://github.com/algebrain/lcmm-observe).

Если позже появится реальная потребность в более глубоком техническом слое,
это можно будет добавить отдельно.

Для базовой совместимости с
[`LCMM`](https://github.com/algebrain/lcmm-docs) это не требуется.

 ## 7.1 Данные о запуске как часть диагностики

  Если проект делает запуск приложения явной частью app-level сборки, сведения о
  ходе запуска могут стать полезной частью support/debug сценария.

  Минимально полезная картина здесь такая:

  - сведения о запуске помогают понять, на какой фазе оборвалась сборка;
  - в эти сведения не должны попадать секреты или избыточная телеметрия;
  - это не отдельная система наблюдаемости, а короткая диагностическая сводка
    уровня приложения.

  Важно удерживать правильный акцент:

  - подробная диагностика запуска допустима;
  - но она не должна разрастаться раньше, чем проект действительно начинает в ней
    нуждаться.

## 8. Минимальный итог

Для `LCMF` достаточно такого минимального набора:

- не терять `correlation-id` и `request-id`;
- продолжать причинно-следственную цепочку в `bus`;
- уметь оставить короткий безопасный технический след;
- уметь собирать безопасный support/debug payload.

Этого достаточно, чтобы связать frontend-проблему с backend trace, не
превращая ранний `LCMF` в тяжелую observability-подсистему.
