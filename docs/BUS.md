# Документация: Frontend message bus

Этот документ описывает `bus` в `LCMF` и его обычное место в приложении.

Для общего порядка app-level сборки см. [`APP.md`](./APP.md).
Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для формы отдельного модуля см. [`MODULE.md`](./MODULE.md).

## 1. Главная мысль

В `LCMF` шина является главным механизмом слабой связанности между модулями.

Это особенно важно потому, что `LCMF` строится не из нескольких крупных
подсистем, а из большого числа маленьких модулей.

Шина нужна, чтобы:

- маленькие модули не срастались прямыми зависимостями;
- межмодульная координация оставалась явной;
- добавление и изменение модулей не требовало знания их внутренних деталей;
- общая картина системы читалась как поток предметных фактов, а не как сеть
  прямых вызовов.

Коротко:

- `bus` — главный межмодульный канал;
- `registry` — вспомогательный канал для узких случаев синхронного чтения;
- прямое знание чужих внутренностей не допускается.

## 2. Где шина живет

В обычном `LCMF`-приложении шина создается не внутри модуля, а на app-level.

Обычная картина такая:

- приложение создает одну общую шину;
- приложение передает ее в модули через `init!`;
- модули публикуют свои сообщения в эту шину;
- другие модули реагируют на нужные им сообщения;
- если обработчик публикует следующее сообщение, причинно-следственная цепочка
  продолжается.

Простой app-level пример:

```clojure
(ns app.system
  (:require [lcmf.bus :as bus]))

(defn make-system []
  {:bus (bus/make-bus)})
```

Смысл здесь простой:

- шина общая;
- модуль не поднимает свою отдельную шину;
- предметная координация идет через один app-level канал.

## 3. Почему именно `bus`

Для `LCMF` это означает следующее:

- модуль не координируется с другими модулями через прямые вызовы;
- модуль не читает чужое состояние как основной способ взаимодействия;
- `registry` не заменяет шину;
- межмодульная картина системы в первую очередь читается как поток сообщений.

Если возникает ощущение, что основная координация идет:

- через прямые вызовы;
- через общий store;
- через `registry`;

то это архитектурный сигнал вернуться к исходному замыслу `LCMF`.

## 4. Какие сообщения живут в шине

В `LCMF` полезно различать два класса сообщений:

- предметные сообщения модулей;
- технические сообщения app-level слоя.

Примеры предметных сообщений:

- `:booking/created`
- `:accounts/signed-out`
- `:catalog/slot-closed`

Примеры app-level технических сообщений:

- `:app/reconnected`
- `:app/external-message-received`

Это различие важно, потому что:

- модуль не должен публиковать app-level технические сообщения от имени
  приложения;
- приложение не должно публиковать предметные сообщения от имени модуля;
- не каждое изменение интерфейса вообще должно становиться сообщением шины.

Шина в `LCMF` нужна прежде всего для предметной межмодульной координации.

## 5. Как начинается и продолжается цепочка сообщений

Для обычного приложения важно понимать не только отдельные вызовы API, но и
общий ход сообщений.

### 5.1 Первое сообщение

Первое сообщение в цепочке вполне может появиться по итогам модульной
операции, связанной с HTTP.

```clojure
(defn load-bookings!
  [{:keys [http bus]}]
  (-> (http/request! http {:method :get :path "/bookings"})
      (.then (fn [bookings]
               (bus/publish! bus
                             :booking/list-loaded
                             {:count (count bookings)}
                             {:module :booking})
               bookings))))
```

Что здесь важно:

- модуль сам публикует предметный факт;
- дальше другие модули могут реагировать на него без прямой связи с `booking`;
- сообщение становится частью общей межмодульной цепочки.

### 5.2 Следующее сообщение

Если модуль реагирует на сообщение и публикует следующее, он не должен начинать
новую цепочку заново.

```clojure
(defn on-booking-created [bus envelope]
  (bus/publish! bus
                :notify/booking-created
                {:booking-id (get-in envelope [:payload :id])}
                {:parent-envelope envelope
                 :module :notify}))
```

Смысл `:parent-envelope` такой:

- сохраняется тот же `correlation-id`, если он уже есть;
- причинно-следственная цепочка продолжает расти;
- позже можно понять, какое сообщение породило следующее.

## 6. Минимальная форма конверта

Даже простая первая версия шины должна работать не с сырой нагрузкой, а с
конвертом сообщения.

Минимальная practically useful форма:

```clojure
{:event-type :booking/created
 :module :booking
 :payload {:id "b-1" :user-id "u-alice" :slot-id "slot-09-00"}
 :message-id "..."
 :created-at 1710000000}
```

Желательно иметь также:

- `:correlation-id`
- данные, позволяющие продолжать причинно-следственную цепочку

Здесь важна не техническая роскошь, а дисциплина:

- шина работает с конвертами;
- модуль всегда указывает `:module`;
- следующее сообщение должно продолжать уже начатую цепочку.

## 7. Основные функции

Для первой версии `LCMF`-шины достаточно очень небольшого публичного API:

- `make-bus`
- `publish!`
- `subscribe!`
- `unsubscribe!`

### `make-bus`

Создает общую шину приложения.

```clojure
(def app-bus
  (bus/make-bus))
```

### `subscribe!`

Подписывает обработчик на событие.

Общая форма:

```clojure
(subscribe! bus event-type handler)
(subscribe! bus event-type handler opts)
```

Где `handler` обычно имеет вид:

```clojure
(fn [bus envelope] ...)
```

### `publish!`

Публикует сообщение в шину.

Общая форма:

```clojure
(publish! bus event-type payload opts)
```

В `opts` важны:

- `:module`
- `:parent-envelope`, если сообщение публикуется в ответ на другое

### `unsubscribe!`

Снимает подписку.

Общая форма:

```clojure
(unsubscribe! bus event-type matcher)
```

Для архитектурного документа этого достаточно.
Все дополнительные вспомогательные возможности являются вторичными.

## 8. Что важно не делать

В первой версии `LCMF`-шины особенно важно не делать следующих вещей:

- не строить межмодульную координацию на прямых вызовах вместо шины;
- не превращать `registry` в замену `bus`;
- не публиковать сообщения без `:module`;
- не терять причинно-следственную цепочку при публикации следующего сообщения;
- не превращать шину в общий канал для любых мелких изменений интерфейса;
- не привязывать архитектурную идею шины к конкретной браузерной технике как
  к ее смыслу.

## 9. Минимальный итог

Шина в `LCMF` — это главный механизм слабой связанности между модулями.

Ее задача — не добавить "еще один уровень абстракции", а сделать жизнеспособной
систему из большого числа маленьких модулей.

С самого начала важно удержать правильную форму:

- одна общая app-level шина;
- модули публикуют и потребляют сообщения через нее;
- сообщения идут в виде конвертов;
- `registry` остается вспомогательным механизмом;
- шина описывает межмодульную предметную координацию, а не произвольную жизнь
  интерфейса.
