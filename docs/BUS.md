# Документация: Frontend message bus

Этот документ описывает frontend `bus` в `LCMF` и его обычное место в
приложении.

Для общего порядка app-level сборки см. [`APP.md`](./APP.md).
Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для формы отдельного модуля см. [`MODULE.md`](./MODULE.md).

Первая версия `LCMF`-шины может быть очень простой. На этом этапе достаточно
тонкой обертки над `document.dispatchEvent(new CustomEvent(...))`, если внешний
API и архитектурная дисциплина остаются под контролем.

## 1. Где шина живет в обычном приложении

В обычном `LCMF`-приложении шина создается не внутри модуля, а на app-level.

Обычная картина такая:

- приложение создает одну общую шину;
- приложение передает ее в `LCMF`-модули через `init!`;
- модуль публикует свои сообщения в эту шину;
- другие модули подписываются на нужные им сообщения;
- причинно-следственная цепочка продолжается через `parent-envelope`, если
  модуль публикует следующее сообщение в ответ на уже полученное.

Небольшой пример:

Это app-level пример. Он показывает создание общей шины в приложении.

```clojure
(ns app.system
  (:require [lcmf.bus :as bus]))

(defn make-system []
  (let [app-bus (bus/make-bus)]
    {:bus app-bus}))
```

Смысл этого примера простой:

- шина создается в корне приложения;
- модули получают уже готовую шину;
- каждый модуль работает с одной общей шиной, а не поднимает свою отдельную.

## 2. Почему именно `bus`

Для `LCMF` шина является главным механизмом слабой связанности.

Это означает:

- модуль не координируется с другими модулями через прямые вызовы;
- модуль не читает чужое состояние как основной способ взаимодействия;
- публичные синхронные чтения через `registry` остаются вспомогательным
  механизмом для узких read-case;
- межмодульная картина системы в первую очередь читается как поток сообщений.

Если в приложении складывается ощущение, что главная координация идет через
store или через `registry`, а не через шину, это архитектурный сигнал
вернуться к исходному замыслу `LCMF`.

## 3. Быстрый старт

Минимальный рабочий пример:

Это пример использования самой `LCMF`-шины как библиотеки, без привязки к
конкретному уровню приложения или модуля.

```clojure
(require '[lcmf.bus :as bus])

(def b (bus/make-bus))

(bus/subscribe! b
                :demo/ping
                (fn [_ envelope]
                  (js/console.log "Got" (clj->js (:payload envelope)))))

(bus/publish! b
              :demo/ping
              {:msg "hello"}
              {:module :demo})
```

В первой версии этого достаточно:

- `publish!` строит конверт сообщения;
- `subscribe!` подписывает обработчик на `event-type`;
- шина доставляет сообщение всем подходящим подписчикам.

## 4. Как начинается и продолжается цепочка сообщений

Для обычного приложения важно понимать не только отдельные вызовы API, но и
общий ход сообщений.

### 4.1 Первое сообщение из HTTP-операции модуля

HTTP-операция frontend-модуля вполне может быть началом цепочки.

Это пример уровня `LCMF`-модуля.

```clojure
(defn load-bookings!
  [{:keys [http bus]}]
  (-> (http/request! http {:method :get :path "/bookings"})
      (.then (fn [bookings]
               (bus/publish! bus
                             :booking/list-loaded
                             {:count (count bookings)}
                             {:module :booking})
               bookings))))
```

Что здесь важно:

- модуль сам публикует сообщение по результату своей операции;
- дальше другие модули могут на него реагировать без прямой связи с `booking`;
- сообщение становится частью общей межмодульной цепочки.

### 4.2 Следующее сообщение из обработчика модуля

Если модуль реагирует на сообщение и публикует следующее, он не должен начинать
новую цепочку заново.

Это пример обработчика уровня `LCMF`-модуля.

```clojure
(defn on-booking-created [bus envelope]
  (bus/publish! bus
                :notify/booking-created
                {:booking-id (get-in envelope [:payload :id])}
                {:parent-envelope envelope
                 :module :notify}))
```

Смысл `:parent-envelope` такой:

- сохраняется тот же `correlation-id`, если он уже есть;
- причинно-следственная цепочка продолжает расти;
- позже можно понять, какое сообщение породило следующее.

## 5. Конструктор

### `make-bus`

Создает новый экземпляр frontend-шины.

```clojure
(require '[lcmf.bus :as bus])

(def app-bus
  (bus/make-bus))
```

Первая реализация может использовать:

- `document.addEventListener`
- `document.removeEventListener`
- `document.dispatchEvent`
- `CustomEvent`

Важно:

- внешний API `LCMF`-шины не должен сводиться к прямому использованию DOM API;
- приложение и модули должны работать с `make-bus`, `publish!`,
  `subscribe!` и `unsubscribe!`, а не с `document.dispatchEvent` напрямую;
- если реализация позже изменится, публичный контракт шины должен остаться
  тем же.

### Пример простой реализации

Это пример уровня библиотеки `lcmf.bus`, а не пример кода приложения или
модуля.

```clojure
(ns lcmf.bus)

(defn make-bus []
  {:target js/document})
```

Это только минимальный эскиз. Полезная часть архитектуры находится не в
сложности объекта, а в дисциплине публикации и подписки.

## 6. Форма конверта

Даже в простой первой версии шина должна работать не с сырой нагрузкой, а с
конвертом сообщения.

Минимальный shape:

Это пример формы конверта шины, то есть shape данных библиотеки, а не код
конкретного `LCMF`-модуля.

```clojure
{:event-type :booking/created
 :module :booking
 :payload {:id "b-1" :user-id "u-alice" :slot-id "slot-09-00"}
 :message-id "..."
 :correlation-id "..."
 :parent-message-id "..."
 :created-at 1710000000}
```

Не все поля обязаны быть идеально реализованы уже в первой итерации, но
архитурно важно с самого начала держать именно конверт, а не просто
`event-type + payload`.

Минимальный practically useful набор:

- `:event-type`
- `:module`
- `:payload`
- `:message-id`
- `:created-at`

Желательно иметь также:

- `:correlation-id`
- `:parent-message-id`

## 7. Основные функции

### `subscribe!`

Подписывает обработчик на определенный `event-type`.

Сигнатура:

```clojure
(subscribe! bus event-type handler)
(subscribe! bus event-type handler opts)
```

Где:

- `event-type` — keyword, например `:booking/created`;
- `handler` — функция вида `(fn [bus envelope])`;
- `opts` — дополнительные опции подписки, например `:meta`.

Пример:

Это пример подписки из кода `LCMF`-модуля или app-level runtime.

```clojure
(bus/subscribe! app-bus
                :booking/created
                (fn [bus envelope]
                  (js/console.log "booking created"
                                  (clj->js (:payload envelope)))))
```

### `publish!`

Публикует сообщение в шину.

Сигнатура:

```clojure
(publish! bus event-type payload opts)
```

В `opts` важны:

- `:module` — обязателен;
- `:parent-envelope` — нужен, если сообщение публикуется в ответ на другое.

Пример корневого сообщения:

Это пример публикации из кода `LCMF`-модуля.

```clojure
(bus/publish! app-bus
              :accounts/signed-out
              {:user-id "u-alice"}
              {:module :accounts})
```

Пример следующего сообщения:

Это пример обработчика уровня `LCMF`-модуля, который продолжает уже начатую
цепочку.

```clojure
(fn [bus envelope]
  (bus/publish! bus
                :notify/booking-created
                {:booking-id (get-in envelope [:payload :id])}
                {:parent-envelope envelope
                 :module :notify}))
```

Даже если первая реализация шины очень простая, `publish!` уже должен:

- собирать конверт;
- сохранять `:module`;
- переносить причинно-следственную связь через `:parent-envelope`.

### `unsubscribe!`

Отписывает обработчик от события.

Сигнатура:

```clojure
(unsubscribe! bus event-type matcher)
```

`matcher` может быть либо самой функцией-обработчиком, либо метаданными
подписки, если такой режим будет поддержан реализацией.

Пример:

Это пример уровня `LCMF`-модуля или app-level runtime.

```clojure
(defn my-handler [_ envelope]
  (js/console.log "called" (clj->js envelope)))

(bus/subscribe! app-bus :demo/ping my-handler)
(bus/unsubscribe! app-bus :demo/ping my-handler)
```

### `listener-count`

Возвращает число подписчиков, если реализация поддерживает такой отладочный
интерфейс.

Это не обязательная бизнес-функция, но полезный инструмент для:

- отладки;
- тестов;
- проверки утечек подписок.

## 8. Предметные и технические сообщения

В `LCMF` полезно различать два класса сообщений:

- предметные сообщения модулей;
- технические сообщения app-level слоя.

Примеры предметных сообщений:

- `:booking/created`
- `:accounts/signed-out`
- `:catalog/slot-closed`

Примеры технических сообщений:

- `:app/reconnected`
- `:app/external-message-received`
- `:app/http-request-failed`

Это различие важно удерживать с самого начала. App-level слой не должен
публиковать предметные сообщения от имени модуля, а модуль не должен
перехватывать ownership app-level технических сообщений.

## 9. Жизненный цикл подписок

Подписка в `LCMF` не должна быть бесконтрольной.

Практические правила:

- приложение и модуль должны понимать, когда подписка создается;
- нужно понимать, когда подписка снимается;
- долгоживущие подписки app-level runtime и модульные подписки нужно отличать
  от короткоживущих подписок экрана или сценария;
- утечки подписок должны быть видны в тестах и отладке.

Если модуль или экран создает подписки при каждом повторном входе и не снимает
их, это приведет не только к утечкам памяти, но и к повторной доставке одного и
того же сообщения нескольким старым обработчикам.

## 10. Что важно не делать

В первой версии `LCMF`-шины важно не делать следующих вещей:

- не давать модулям прямой зависимости от `document.dispatchEvent`;
- не строить межмодульную координацию на прямых вызовах вместо шины;
- не превращать `registry` в замену `bus`;
- не публиковать сообщения без `:module`;
- не терять причинно-следственную цепочку при публикации следующего сообщения.

## 11. Минимальный итог

Шина в `LCMF` — это главный механизм слабой связанности между модулями.

Даже если первая реализация очень проста и держится на `CustomEvent`, важно с
самого начала удержать правильную архитектурную форму:

- одна app-level шина;
- модули публикуют и потребляют сообщения через нее;
- сообщения идут в виде конвертов;
- `registry` остается вспомогательным механизмом, а не конкурентом `bus`.
