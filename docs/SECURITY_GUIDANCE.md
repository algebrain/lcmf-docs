# Документация: Security guidance для frontend

Этот документ описывает, как frontend в `LCMF` должен вести себя рядом с
backend, построенным по принципам
[`LCMM`](https://github.com/algebrain/lcmm-docs).

Речь здесь не о том, как приложение реализует защиту внутри себя. Это задача
app-level слоя backend и его security middleware. Здесь фиксируется другое:

- какие security-сценарии клиент должен считать штатными;
- как frontend должен реагировать на `401`, `403`, `429` и `503`;
- какие ограничения безопасности нельзя обходить клиентскими приемами;
- какие данные нельзя утаскивать в логи, debug payload и bug report.

Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для HTTP-поведения клиента см. [`HTTP.md`](./HTTP.md).
Для realtime transport см. [`WEBSOCKET.md`](./WEBSOCKET.md).
Для трассировки и support/debug payload см.
[`TRACE_AND_OBSERVABILITY.md`](./TRACE_AND_OBSERVABILITY.md).

## 1. Зачем нужен этот документ

В `LCMM` защита живет прежде всего на app-level:

- auth и сессия проверяются на уровне приложения;
- rate limit, временный ban и guard-поведение тоже принадлежат приложению;
- наружу отдается единый безопасный error-contract;
- модуль не должен придумывать собственную несовместимую security-модель.

Для frontend из этого следует важная практическая вещь:

- клиент не должен считать security-ответы случайной поломкой;
- клиент не должен ждать от backend лишних подробностей;
- клиент не должен строить UX на попытке угадать внутреннюю причину отказа.

Этот документ нужен именно для того, чтобы `LCMF`-frontend корректно жил рядом
с app-level защитой backend.

## 2. Что клиент должен считать нормальным security-поведением backend

Для frontend в `LCMF` нормальна такая картина:

- `401` сигнализирует отсутствие или потерю аутентификации;
- `403` сигнализирует запрет доступа;
- `429` сигнализирует ограничение частоты, временное ограничение или ban;
- `503` может означать временную деградацию защитного или инфраструктурного
  слоя;
- auth-ошибка может быть намеренно минималистичной;
- сервер может не раскрывать, существует ли пользователь, ресурс или чужая
  подписка;
- websocket handshake и `subscribe` тоже могут быть отклонены как часть
  штатной политики безопасности.

Практический смысл:

- security-ответы должны быть отдельными UI-состояниями;
- frontend не должен воспринимать их как невозможные случаи;
- клиентская логика должна опираться на `status` и `code`, а не на текст
  `message`.

## 3. `401`, `403`, `429` и `503` как штатные сценарии

Для `LCMF` practically useful базовая картина такая:

- `401` означает, что нужен повторный auth flow, а не бесконечный retry того же
  действия;
- `403` означает, что действие понято, но запрещено политикой доступа;
- `429` означает, что нужен backoff, cooldown или пауза перед новой попыткой;
- `503` означает временную деградацию, при которой нужен контролируемый retry,
  а не паническое повторение запросов.

Практические правила:

- не повторяйте автоматически чувствительные действия после `401`;
- не маскируйте `403` под сетевую ошибку;
- не спамьте повторными запросами после `429`;
- даже при `503` число автоматических повторов должно быть ограничено;
- если backend передал `Retry-After`, клиент должен учитывать его как базовую
  задержку.

## 4. Anti-enumeration и безопасные auth-ошибки

Backend по принципам `LCMM` может намеренно скрывать детали auth-ошибки.

Для frontend это означает:

- нельзя ожидать, что сервер скажет, существует ли пользователь;
- нельзя ожидать разных текстов для "неверный логин" и "неверный пароль";
- нельзя строить UX на различии безопасных auth-ошибок;
- нельзя считать отказ в websocket-подписке источником лишней предметной
  информации.

Практический вывод:

- для auth-ошибок обычно нужен единый безопасный текст;
- клиентская логика должна ветвиться по устойчивым признакам ответа;
- если подробностей нет, это не недостаток контракта, а часть security-модели.

## 5. App-level auth и поведение защищенных экранов

Поскольку auth принадлежит app-level слою backend, frontend тоже должен
реагировать на него как на app-level состояние.

Для клиента это обычно означает:

- защищенный экран должен уметь перейти в состояние "нужен повторный вход";
- модульный UI не должен дублировать собственную auth-модель поверх app-level
  поведения;
- потеря сессии не должна выглядеть как произвольная поломка конкретного
  модуля.

Практические правила:

- при `401` останавливайте текущий protected flow;
- переводите экран или приложение в понятное состояние повторной
  аутентификации;
- не повторяйте автоматически state-changing действие без явного контроля после
  повторного входа.

## 6. Ограничение частоты, временные блокировки и cooldown

Для `LCMM` repeated invalid requests не считаются безобидным циклом. Они могут
попасть под rate limit, временный ban или другой guard-сценарий.

Для frontend это означает:

- repeated invalid login attempts нельзя автоматически крутить без паузы;
- repeated retries после `429` могут только усугубить ситуацию;
- интерфейсу полезно явно показывать временное ограничение, а не скрывать его.

Практически клиенту полезно:

- временно отключать кнопку повторной отправки после `429`;
- использовать `Retry-After`, если он есть;
- если `Retry-After` нет, вводить короткий локальный cooldown;
- не запускать бесконечный мгновенный reconnect или retry loop.

## 7. Redaction, debug payload и bug report

Для `LCMF` frontend должен помогать диагностике, но не ценой утечки секретов.

Минимально опасные данные, которые нельзя утаскивать в клиентские логи,
support/debug payload и bug report:

- access token;
- cookie;
- session id;
- `authorization` header;
- поля `password`, `token`, `secret`, `api-key`;
- сырой payload целиком, если он содержит чувствительные данные.

Практически это означает:

- в debug payload сохраняйте `status`, `code`, `correlation-id`,
  `request-id`, endpoint и время;
- не прикладывайте секреты "для удобства разбора";
- при клиентском логировании применяйте такую же redaction-дисциплину, как и в
  app-level логах backend.

## 8. WebSocket и security-сценарии

В `LCMF` websocket остается app-level transport-слоем и подчиняется общей
security-политике приложения.

Для frontend это означает:

- handshake может быть отклонен;
- `Origin` может проверяться;
- каждая подписка может отдельно авторизовываться;
- reconnect не гарантирует автоматического успеха, если политика доступа не
  выполняется;
- отказ в подписке не обязан раскрывать внутреннюю причину.

Практические правила:

- отказ в websocket-подписке считайте штатным сценарием;
- не пытайтесь угадывать по тексту отказа существование чужого ресурса;
- после reconnect будьте готовы и к повторному отказу;
- если соединение закрыто или отклонено, переходите к сценарию resync по HTTP,
  а не к бесконечному агрессивному переподключению.

## 9. Что frontend не должен делать

Frontend в `LCMF` не должен:

- ожидать подробной auth-ошибки там, где backend сознательно дает безопасный
  минимум;
- строить разные UX-ветки по догадкам о существовании пользователя;
- автоматически повторять `429` как обычный сетевой сбой;
- передавать секреты в URL и query string;
- прикладывать cookie, session id или token к bug report;
- считать websocket-подписку всегда успешной;
- проектировать reconnect как бесконечный tight loop без cooldown и resync.

## 10. Минимальный итог

Для `LCMF` practically useful security baseline на стороне frontend выглядит
так:

- `401`, `403`, `429` и `503` считаются частью штатного поведения системы;
- клиент опирается на `status` и `code`, а не на подробные тексты;
- anti-enumeration и безопасные отказы важнее удобства угадывания;
- rate limit и временные блокировки требуют паузы, а не агрессивного retry;
- debug и support flow должны сохранять trace id, но не должны утекать
  секретами.
