# Документация: Безопасное поведение frontend

Этот документ описывает, как frontend в `LCMF` должен вести себя рядом с
backend, построенным по принципам
[`LCMM`](https://github.com/algebrain/lcmm-docs).

Речь идет о безопасном поведении клиента в обычной работе:

- какие security-сценарии клиент должен считать штатными;
- как frontend должен реагировать на `401`, `403`, `429` и `503`;
- как не ломать защитную модель backend своими догадками;
- какие данные нельзя уносить в логи, debug payload и bug report.

Внутреннюю защиту внутри себя клиентский модуль здесь не строит.
Auth, guard и app-level security policy принадлежат приложению и backend-слою.

Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для HTTP-поведения клиента см. [`HTTP.md`](./HTTP.md).
Для realtime transport см. [`WEBSOCKET.md`](./WEBSOCKET.md).
Для трассировки и support/debug payload см.
[`TRACE_AND_OBSERVABILITY.md`](./TRACE_AND_OBSERVABILITY.md).

Важно держать этот текст в согласии с backend-ориентирами:

## 1. Зачем нужен этот документ

В `LCMM` защита живет прежде всего на app-level:

- auth и сессия проверяются на уровне приложения;
- ограничение частоты, временные блокировки и guard-поведение тоже принадлежат
  приложению;
- наружу отдается единый безопасный error-contract;
- модуль не строит собственную несовместимую security-модель.

Для frontend из этого следует практический вывод:

- клиент должен считать security-ответы нормальной частью системы;
- клиент должен спокойно переживать защитные отказы;
- клиент не должен строить UX на попытке угадать скрытую внутреннюю причину.

Именно для этого и нужен этот документ.

## 2. Что клиент должен считать нормальным поведением backend

Для frontend в `LCMF` нормальна такая картина:

- `401` означает отсутствие или потерю аутентификации;
- `403` означает запрет доступа;
- `429` означает временное ограничение и необходимость паузы;
- `503` означает временную деградацию, при которой нужен спокойный режим
  восстановления;
- auth-ошибка может быть намеренно минималистичной;
- сервер может не раскрывать, существует ли пользователь, ресурс или чужая
  подписка;
- websocket handshake и `subscribe` тоже могут быть отклонены как часть
  штатной политики безопасности.

Практический смысл:

- security-ответы должны быть отдельными UI-состояниями;
- frontend не должен воспринимать их как невозможные случаи;
- клиентская логика должна опираться на `status` и `code`, а не на текст
  `message`.

## 3. `401`, `403`, `429` и `503` как разные пользовательские режимы

Для клиента эти статусы важны не только как коды ответа, но и как разные
режимы поведения.

### `401`

Это сигнал, что приложение потеряло текущий auth-state или не имеет его.

Практический смысл:

- protected flow нужно остановить;
- экран или приложение нужно перевести в состояние повторной аутентификации;
- старое действие не нужно автоматически продолжать без нового явного шага.

### `403`

Это сигнал, что действие понято, но запрещено политикой доступа.

Практический смысл:

- не нужно запускать новый login flow;
- нужно честно показать запрет;
- не нужно маскировать такой ответ под обычный сетевой сбой.

### `429`

Это сигнал, что сейчас нужна пауза.

Практический смысл:

- не нужно немедленно повторять попытку;
- если сервер дал `Retry-After`, его нужно учитывать;
- если точного указания нет, проекту нужен спокойный режим охлаждения, а не
  мгновенный новый цикл.

### `503`

Это сигнал временной деградации.

Практический смысл:

- клиенту нужен контролируемый режим ожидания и восстановления;
- число автоматических повторов должно быть ограничено;
- пользователь не должен видеть хаотичное "дергание" интерфейса.

## 4. Anti-enumeration и безопасные auth-ошибки

Backend по принципам `LCMM` может намеренно скрывать детали auth-ошибки.

Для frontend это означает:

- нельзя ожидать, что сервер скажет, существует ли пользователь;
- нельзя ожидать разных текстов для "неверный логин" и "неверный пароль";
- нельзя строить UX на различии безопасных auth-ошибок;
- нельзя считать отказ в websocket-подписке источником лишней предметной
  информации.

Практический вывод:

- для auth-ошибок обычно нужен единый безопасный текст;
- клиентская логика должна ветвиться по устойчивым признакам ответа;
- если подробностей нет, это часть security-модели, а не недостаток
  контракта.

Frontend не должен пытаться "улучшить" такую модель своими догадками.

Это означает:

- не придумывать скрытые причины отказа по косвенным признакам;
- не показывать пользователю лишние выводы, которых сервер не подтвердил;
- не строить разные ветки поведения на предположении о внутренних причинах
  отказа.

## 5. Auth как app-level состояние

Поскольку auth принадлежит app-level слою backend, frontend тоже должен
реагировать на него как на app-level состояние.

Для клиента это обычно означает:

- защищенный экран умеет переходить в состояние "нужен повторный вход";
- модульный UI не дублирует собственную auth-модель поверх общего поведения;
- потеря сессии не выглядит как поломка конкретного предметного модуля.

Практические правила:

- при `401` останавливайте текущий защищенный сценарий;
- переводите экран или приложение в понятное состояние повторной
  аутентификации;
- судьбу прерванного действия решайте отдельно;
- по умолчанию безопаснее ждать нового явного шага пользователя, чем тихо
  доигрывать старое действие после повторного входа.

## 6. Ограничение частоты и временные блокировки

Для `LCMM` repeated invalid requests не считаются безобидным циклом.
Они могут попасть под ограничение частоты, временную блокировку или другой
guard-сценарий.

Для frontend это означает:

- повторяющиеся неудачные login attempts нельзя крутить без паузы;
- повторяющиеся попытки после `429` могут только ухудшить ситуацию;
- интерфейсу полезно явно показывать временное ограничение.

Практический инвариант здесь такой:

- после `429` у клиента должен быть режим паузы;
- клиент не должен продолжать попытки немедленно;
- если сервер дал `Retry-After`, его нужно уважать;
- конкретный способ реализации этой паузы остается за проектом.

Это может быть:

- временная блокировка кнопки;
- локальный таймер;
- app-level состояние охлаждения;
- helper в HTTP runtime;
- другой спокойный и понятный пользователю способ.

## 7. WebSocket и security-сценарии

В `LCMF` websocket остается app-level transport-слоем и подчиняется общей
security-политике приложения.

Для frontend это означает:

- handshake может быть отклонен;
- `Origin` может проверяться;
- каждая подписка может отдельно авторизовываться;
- reconnect не гарантирует автоматического успеха;
- отказ в подписке не обязан раскрывать внутреннюю причину.

Практические правила:

- отказ в websocket-подписке считайте штатным сценарием;
- не пытайтесь угадывать по тексту отказа существование чужого ресурса;
- после reconnect будьте готовы и к повторному отказу;
- если соединение закрыто или отклонено, переходите к спокойному сценарию
  resync по HTTP, а не к бесконечному агрессивному переподключению.

## 8. Безопасный debug и support flow

Frontend должен помогать диагностике и при этом сохранять безопасное поведение.

Минимально полезный набор данных для debug/support:

- `status`;
- `code`;
- `correlation-id`;
- `request-id`, если он доступен;
- endpoint или screen name;
- время ошибки.

Минимально опасные данные, которые нельзя уносить в клиентские логи,
support/debug payload и bug report:

- access token;
- cookie;
- session id;
- `authorization` header;
- поля `password`, `token`, `secret`, `api-key`;
- сырой payload целиком, если он содержит чувствительные данные.

Практический вывод:

- frontend должен сохранять trace id и безопасный контекст ошибки;
- frontend не должен ради удобства разбора утаскивать секреты.

## 9. Что frontend не должен делать

Frontend в `LCMF` не должен:

- ждать подробной auth-ошибки там, где backend сознательно дает безопасный
  минимум;
- строить UX-ветки по догадкам о существовании пользователя;
- автоматически повторять `401` как будто это обычный временный сбой;
- автоматически повторять `429` как будто это обычный сетевой сбой;
- передавать секреты в URL и query string;
- прикладывать cookie, session id или token к bug report;
- считать websocket-подписку всегда успешной;
- проектировать reconnect как бесконечный тесный цикл без паузы и resync.

## 10. Минимальный итог

Для `LCMF` безопасное поведение frontend на старте выглядит так:

- `401`, `403`, `429` и `503` считаются частью штатного поведения системы;
- клиент опирается на `status` и `code`, а не на подробные тексты;
- anti-enumeration и безопасные отказы важнее удобства угадывания;
- временные ограничения требуют паузы, а не агрессивного повтора;
- debug и support flow сохраняют trace id, но не уносят секреты.

Этого достаточно, чтобы frontend корректно жил рядом с app-level защитой
backend, не перетаскивая в себя внутреннюю механику security-layer.
