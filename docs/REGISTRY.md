# Документация: Реестр публичных чтений

Этот документ описывает `registry` в `LCMF`.

Речь идет о механизме, который позволяет одному модулю получить чужие публичные
данные без прямой связанности с внутренней формой состояния другого модуля.

Этот документ также задает роль библиотеки `lcmf-registry`.

Реализация библиотеки реестра:
[`lcmf-registry`](https://github.com/algebrain/lcmf-registry).

Описание интерфейса библиотеки:
[`lcmf-registry/docs/INTERFACE.md`](https://github.com/algebrain/lcmf-registry/blob/main/docs/INTERFACE.md).

Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для формы отдельного модуля см. [`MODULE.md`](./MODULE.md).
Для app-level слоя, который создает реестр и выполняет проверку связей, см.
[`APP.md`](./APP.md).

## 1. Главная мысль

`registry` в `LCMF` — это узкий контрактный канал синхронного чтения между
модулями.

Он нужен не затем, чтобы стать вторым главным механизмом системы.
Он нужен затем, чтобы системе из многих маленьких модулей иногда было удобно
сделать короткое публичное чтение без:

- прямого знания чужих внутренностей;
- переноса всей координации в шину;
- превращения общего состояния в неявный общий склад данных.

Коротко:

- `bus` координирует;
- `registry` дает узкое синхронное чтение;
- `registry` не должен занимать место шины.

## 2. Когда использовать `registry`

Используйте `registry`, когда:

- нужно синхронное чтение данных другого `LCMF`-модуля;
- зависимость должна быть явной и проверяемой;
- для этого случая не нужна полноценная цепочка сообщений;
- нужен конкретный внешний сценарий чтения, а не доступ к чужому состоянию вообще.

Не используйте `registry` для:

- основной межмодульной координации;
- побочных эффектов и реактивных процессов;
- обхода внутренних границ другого модуля;
- каскадных вызовов provider -> provider;
- превращения чужого модуля в "место, куда можно сходить за любыми данными".

В `LCMF` важно держать в голове общую рамку:

- `registry` — это узкий инструмент;
- `registry` вторичен по отношению к `bus`;
- если через `registry` начинает проходить все больше координации, это
  архитектурный сигнал пересмотреть границы модулей или сам сценарий
  взаимодействия.

## 3. Зачем он особенно полезен для маленьких модулей

Поскольку `LCMF` строится из большого числа маленьких модулей, системе нужен
способ иногда делать короткое чтение чужих публичных данных без прямого
срастания.

Например, модулю может понадобиться узнать:

- существует ли пользователь;
- открыт ли слот;
- какой публичный статус сейчас у другой сущности.

Для таких случаев `registry` полезен, потому что позволяет:

- оставить модуль маленьким;
- не тащить знание чужой внутренней формы;
- не строить избыточную цепочку сообщений там, где нужен именно сценарий чтения.

Но этот плюс легко превращается в риск:

- чем удобнее `registry`, тем сильнее соблазн использовать его слишком широко;
- поэтому границы его применения нужно удерживать особенно жестко.

## 4. Уровни применения

Реестр применяется на трех уровнях:

- app-level слой создает `registry` и выполняет проверку связей;
- модуль-владелец регистрирует свои публичные чтения;
- модуль-потребитель объявляет обязательные зависимости и получает provider для
  вызова.

Обычная картина такая:

- приложение создает один общий `registry`;
- модули-владельцы регистрируют свои providers;
- модули-потребители объявляют обязательные чтения;
- приложение выполняет раннюю проверку связей;
- после этого provider используется внутри нужного модуля.

Здесь важен акцент:

- `registry` прежде всего межмодульный механизм;
- app-level слой не должен превращать его в удобный общий центр предметной
  сборки.

## 5. Основные функции

Для первой practically useful версии достаточно такого набора:

- `make-registry`
- `register-provider!`
- `resolve-provider`, если нужен мягкий режим
- `require-provider`
- `declare-requirements!`
- `validate-requirements`, если нужен явный диагностический шаг
- `assert-requirements!`

### `make-registry`

Создает общий in-memory реестр.

```clojure
(def registry
  (registry/make-registry))
```

Это app-level операция.
Приложение создает один общий `registry` на сборку системы.

### `register-provider!`

Регистрирует публичное чтение модуля с уникальным `provider-id`.

```clojure
(registry/register-provider!
 registry
 {:provider-id :accounts/get-by-id
  :module :accounts
  :provider-fn (fn [{:keys [user-id]}]
                 (when-let [user (get-in @accounts-state [:users user-id])]
                   {:id (:id user)
                    :login (:login user)
                    :role (:role user)}))})
```

Это операция модуля-владельца.

### `require-provider`

Возвращает provider-функцию, а при отсутствии сразу сигнализирует об ошибке.

```clojure
(let [get-user (registry/require-provider registry :accounts/get-by-id)]
  (get-user {:user-id "u-alice"}))
```

Для `LCMF` это основной выбор для обязательных зависимостей.

Смысл здесь такой:

- если модулю действительно нужен чужой provider;
- это должно быть видно явно;
- отсутствие связи не должно всплывать мутно в случайный момент обычной работы.

### `declare-requirements!`

Объявляет обязательные внешние read-зависимости модуля.

```clojure
(registry/declare-requirements! registry
                                :booking
                                #{:accounts/get-by-id
                                  :catalog/get-slot-by-id})
```

Это обычно делается в `init!` модуля-потребителя.

### `assert-requirements!`

Выполняет раннюю проверку корректности модульных связей до обычной работы
приложения.

```clojure
(registry/assert-requirements! registry)
```

Это app-level операция.

Важно правильно понимать ее смысл на frontend-стороне:

- это не "серверный fail-fast ради строгости";
- это ранняя проверка того, что обязательные модульные связи действительно
  собраны корректно;
- приложение не должно делать вид, что оно собрано нормально, если обязательные
  связи между модулями не закрыты.

## 6. Почему `require-provider` важнее, чем мягкий режим

Во frontend `LCMF` основной акцент должен быть на явных и заранее проверяемых
межмодульных связях.

Поэтому practically useful картина такая:

- обязательные зависимости объявляются явно;
- приложение проверяет их при сборке;
- модуль использует `require-provider` для того, что действительно обязательно.

Мягкий режим поиска provider может существовать как вспомогательная возможность,
но он не должен становиться обычной картиной.

Иначе архитектура постепенно съедет в:

- "если provider есть, хорошо";
- "если нет, как-нибудь переживем";
- "настоящая форма связей не очень понятна заранее".

Для системы из многих маленьких модулей это плохой путь.

## 7. Что должен возвращать provider

Provider проектируется вокруг внешнего сценария чтения, а не вокруг удобства
внутренней модели владельца.

Практические правила:

- provider возвращает минимально достаточные данные;
- provider не раскрывает весь внутренний участок состояния;
- provider не выполняет побочных эффектов;
- provider не превращается в универсальный способ "спросить у модуля что
  угодно";
- provider не должен быть слишком широким и закрывать сразу слишком много
  разных сценариев.

### Пример хорошего provider

```clojure
(registry/register-provider!
 registry
 {:provider-id :accounts/get-by-id
  :module :accounts
  :provider-fn (fn [{:keys [user-id]}]
                 (when-let [user (get-in @accounts-state [:users user-id])]
                   {:id (:id user)
                    :login (:login user)
                    :role (:role user)}))})
```

### Пример плохого provider

```clojure
(registry/register-provider!
 registry
 {:provider-id :accounts/raw-state
  :module :accounts
  :provider-fn (fn [_]
                 @accounts-state)})
```

Во втором варианте модуль раскрывает весь свой внутренний участок состояния и
ломает границу владения.

Плохим provider может быть и более "аккуратный", но слишком широкий вариант,
если через него внешний код начинает читать почти все подряд.

## 8. Связь с `bus`

`registry` не заменяет `bus`.

Это ключевая мысль всего документа.

В `LCMF` роли разделены так:

- `bus` координирует модули;
- `registry` закрывает отдельные случаи синхронного чтения;
- если `registry` начинает выглядеть как главный механизм системы, это ошибка
  архитектурного акцента.

### Пример хорошего разделения ролей

```clojure
;; booking читает публичные данные catalog через registry
(let [get-slot (registry/require-provider registry :catalog/get-slot-by-id)
      slot (get-slot {:slot-id "slot-09-00"})]
  ...)

;; а о результате своей работы сообщает через bus
(bus/publish! bus
              :booking/created
              {:id "b-1" :slot-id "slot-09-00" :user-id "u-alice"}
              {:module :booking})
```

Если это различие начинает стираться, документ нужно перечитать именно отсюда.

## 9. App-level слой и `registry`

App-level слой:

- создает `registry`;
- выполняет проверку обязательных связей;
- передает его модулям.

Но app-level слой не должен:

- собирать через providers предметную картину системы вместо модулей;
- превращать `registry` в удобный общий обход модульных границ;
- накапливать у себя предметную логику через последовательные синхронные чтения.

То есть app-level слой должен управлять механизмом `registry`, но не должен
становиться главным потребителем чужих providers в повседневной предметной
логике.

## 10. Роль `lcmf-registry`

Роль `lcmf-registry` нужно понимать узко и точно.

Ее ожидаемая роль:

- хранить регистрацию providers;
- хранить объявленные обязательные зависимости;
- давать явный и короткий API чтения;
- поддерживать fail-fast проверку на старте;
- оставаться вторичной по отношению к `bus`.

По своему архитектурному объему `lcmf-registry` должна быть ближе к
[`lcmm-registry`](https://github.com/algebrain/lcmm-registry), чем к большой
framework-like подсистеме.

Это не должен быть:

- второй главный столп архитектуры рядом с `bus`;
- "умный слой доступа к данным между модулями";
- большая подсистема с богатой внутренней моделью;
- место, где межмодульная координация незаметно уходит из `bus` в sync-read.

Это должна быть маленькая supporting library для контрактных синхронных чтений.

То есть здесь хороший ориентир такой:

- маленький in-memory механизм;
- явные функции;
- минимум магии;
- максимум предсказуемости.

Если библиотека начинает разрастаться так, будто именно через нее должна
проходить настоящая жизнь системы, это архитектурный сигнал остановиться.

## 11. Пример обычной картины использования

```clojure
(defn init-app! []
  (let [registry (registry/make-registry)]
    (accounts/init! {:registry registry ...})
    (catalog/init! {:registry registry ...})
    (booking/init! {:registry registry ...})
    (registry/assert-requirements! registry)
    {:registry registry}))
```

Что показывает этот пример:

- приложение создает один общий `registry`;
- модули-владельцы регистрируют свои публичные чтения;
- модули-потребители объявляют обязательные зависимости;
- app-level слой выполняет раннюю проверку связей;
- после этого модули могут использовать чужие публичные чтения без знания
  внутренностей.

## 12. Минимальный итог

`registry` в `LCMF` — это узкий контрактный механизм публичных синхронных
чтений.

Он особенно полезен потому, что помогает множеству маленьких модулей делать
короткие публичные сценарии чтения без прямого срастания.

Но его нужно удерживать жестко:

- он вторичен по отношению к `bus`;
- он не должен превращаться в универсальный межмодульный доступ к чужим
  данным;
- provider всегда проектируется под конкретный внешний сценарий чтения, а не как
  маленький запрос "внутрь другого модуля".

`lcmf-registry` должна оставаться маленькой поддерживающей библиотекой для
узких sync-read случаев, а не превращаться в альтернативный центр
межмодульной архитектуры.
