# APP_COMPOSITION: как собирать LCMF-приложение

Этот документ объясняет, как собрать app-level слой `LCMF` в целое
frontend-приложение.

Его роль практическая: он показывает опорную схему сборки, порядок шагов и
форму зависимостей, которые приложение передает модулям.

Если вам нужна общая архитектурная рамка app-level слоя, сначала прочитайте
[`APP.md`](./APP.md).
Если вам нужны детали отдельных частей, переходите в профильные документы:

1. [`BUS.md`](./BUS.md)
2. [`REGISTRY.md`](./REGISTRY.md)
3. [`HTTP.md`](./HTTP.md)
4. [`WEBSOCKET.md`](./WEBSOCKET.md)

6. [`SECURITY_GUIDANCE.md`](./SECURITY_GUIDANCE.md)

## 1. Что такое app-level сборка

App-level сборка в `LCMF` нужна для того, чтобы собрать множество маленьких
модулей в одно приложение.

Её задача:

- создать общую инфраструктуру;
- подготовить общий runtime приложения;
- передать модулям их зависимости;
- вызвать `init!` модулей;
- выполнить стартовые проверки;
- вернуть собранное приложение.

Коротко:

- приложение собирает систему;
- модуль ведет себя сам.

Это также объясняет роль `lcmf-app`.

Реализация библиотеки app-level сборки:
[`lcmf-app`](https://github.com/algebrain/lcmf-app).

Её роль в том, чтобы поддержать app-level сборку там, где реально полезны:

- единообразный порядок сборки;
- аккуратная передача зависимостей;
- дешевые helpers для wiring;
- повторяемая дисциплина подключения модулей.

Её роль не в том, чтобы стать новым архитектурным центром приложения.

## 2. Что обычно собирает приложение

Обычный состав app-level сборки такой:

- конфиг приложения;
- `bus`;
- `registry`;
- HTTP runtime;
- websocket runtime, если приложению нужны realtime-сценарии;
- технический след и отладочные hooks, если проект их использует;
- app-level состояние или иная физическая форма модульного runtime, если она
  нужна выбранной реализации;
- итоговая связка этих частей.

Смысл здесь простой:

- общая инфраструктура создается один раз на уровне приложения;
- модуль получает готовые зависимости и не пересобирает их сам;
- app-level код не забирает у модулей владение их логикой.

## 3. Базовый порядок сборки

Базовый порядок app-level сборки такой:

1. загрузить конфиг;
2. создать `bus`;
3. создать `registry`;
4. создать HTTP runtime;
5. создать websocket runtime, если он нужен;
6. подготовить технические зависимости, если проект ими пользуется;
7. собрать модульный runtime состояния, если он нужен выбранной реализации;
8. вызвать `init!` модулей;
9. выполнить `assert-requirements!` для `registry`;
10. выполнить стартовые проверки;
11. вернуть собранное приложение.

Этот порядок полезен потому, что:

- модули получают уже готовые общие зависимости;
- `registry` готов к регистрации провайдеров до вызова `init!`;
- обязательные sync-read контракты проверяются после инициализации модулей;
- websocket остается app-level транспортом, если он вообще нужен приложению.

## 4. Что именно получает модуль

Модуль получает только те зависимости, которые ему реально нужны.

Минимальная практическая форма часто выглядит так:

```clojure
{:bus bus
 :registry registry
 :state booking-state
 :http http-runtime}
```

Возможное расширение:

```clojure
{:bus bus
 :registry registry
 :state booking-state
 :http http-runtime
 :ws ws-runtime
 :trace trace-runtime
 :config module-config}
```

Здесь важно удерживать три правила:

- модуль получает свой участок состояния или свой runtime;
- модуль не получает весь app-state целиком;
- модуль не получает чужие внутренние данные в обход `bus` и `registry`.

## 5. Канонический skeleton

Ниже не шаблон для буквального копирования, а опорная схема app-level сборки.
Она показывает порядок и форму, а не навязывает точные имена функций.

```clojure
(ns my.app
  (:require [lcmf.bus :as bus]
            [lcmf.http :as http]
            [lcmf.registry :as registry]))

(defn make-app-state []
  {:accounts (atom accounts.state/initial-state)
   :booking (atom booking.state/initial-state)})

(defn init-app! []
  (let [config (load-config)
        event-bus (bus/make-bus)
        read-registry (registry/make-registry)
        http-runtime (http/make-runtime {:base-url "/api"})
        app-state (make-app-state)]

    (accounts/init! {:bus event-bus
                     :registry read-registry
                     :http http-runtime
                     :state (:accounts app-state)})

    (booking/init! {:bus event-bus
                    :registry read-registry
                    :http http-runtime
                    :state (:booking app-state)})

    (registry/assert-requirements! read-registry)

    {:config config
     :bus event-bus
     :registry read-registry
     :http http-runtime
     :state app-state}))
```

Если проект использует websocket, он добавляется сюда же, на app-level.
Если проект использует технический след, он тоже подключается здесь.

## 6. Где здесь место для websocket

Websocket включается в app-level сборку только тогда, когда у приложения есть
реальная задача реального времени.

Тогда приложение:

- создает websocket runtime;
- определяет правила подключения;
- определяет, какие входящие сообщения превращаются в app-level действия;
- определяет, какие app-level события действительно нужно передавать наружу.

Это позволяет сохранить правильную границу:

- модуль не владеет transport-слоем websocket;
- приложение не превращает websocket в внешний дубль `bus`;
- websocket остается узким практическим транспортом для конкретных сценариев.

## 7. Где здесь место для технического следа

App-level сборка может также подключать минимальный технический след.

Обычно это означает:

- продолжение `correlation-id`, если он пришел снаружи;
- чтение и сохранение `request-id` как идентификатора конкретного HTTP-запроса;
- короткие отладочные hooks;
- безопасную передачу технических данных между HTTP, websocket и `bus`.

Смысл этого слоя в том, чтобы облегчить разбор проблем и поддержку приложения.
Смысл не в том, чтобы превращать frontend в тяжелую подсистему наблюдаемости.

## 8. Что app-level сборка должна сохранять

Хорошая app-level сборка сохраняет такие свойства:

- новый модуль подключается дешево;
- общая инфраструктура не дублируется;
- модульные границы остаются ясными;
- `bus` остается главным каналом межмодульной координации;
- `registry` остается узким механизмом публичных sync-read;
- app-level код не накапливает предметную логику.

Эти свойства важнее конкретной физической формы сборки.

## 9. Что app-level сборка не должна подменять

App-level сборка не должна подменять собой модульную архитектуру.

Поэтому она не должна:

- превращать app-state в обход границ модулей;
- забирать у модулей владение их состоянием;
- переносить предметную координацию из модулей в app-level код;
- делать `bus` и `registry` второстепенными на фоне прямых доступов;
- разрастаться до большого framework поверх `LCMF`.

Если подключение нового модуля становится дорогим и требует большого объема
ручной связки, это сигнал упростить app-level соглашения или поддерживающие
библиотеки.

## 10. Минимальный итог

`APP_COMPOSITION` отвечает на практический вопрос: как из модулей собрать одно
`LCMF`-приложение.

Он удерживает четыре главные мысли:

- приложение создает общую инфраструктуру;
- модуль получает только свои зависимости;
- порядок сборки должен быть предсказуемым;
- app-level слой помогает модулям работать вместе, не забирая у них их роль.
