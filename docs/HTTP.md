# Документация: HTTP во frontend

Этот документ описывает роль HTTP в `LCMF`, обычное место HTTP-рантайма в
frontend-приложении и поддерживающую библиотеку `lcmf-http`, которая помогает эту
модель реализовать.

Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для формы отдельного модуля см. [`MODULE.md`](./MODULE.md).
Для app-level слоя, который создает HTTP-рантайм, см. [`APP.md`](./APP.md).
Для типового состояния одного запроса поверх `lcmf-http` см. также
[`REQUEST_STATE.md`](./REQUEST_STATE.md).
Если нужен realtime-транспорт, смотрите также [`WEBSOCKET.md`](./WEBSOCKET.md).

Важно держать этот текст в согласии с backend-ориентиром
[`../lcmm-docs/docs/HTTP.md`](/home/algebrain/src/my/lcmm-docs/docs/HTTP.md).

Реализация библиотеки:

- [`lcmf-http`](https://github.com/algebrain/lcmf-http)
- [Интерфейс `lcmf-http`](https://github.com/algebrain/lcmf-http/blob/main/docs/INTERFACE.md)

Это означает:

- backend дает основной HTTP-контракт;
- frontend не придумывает другой параллельный контракт;
- `LCMF` описывает, как этот контракт практически проживается на
  клиенте.

## 1. Назначение

HTTP в `LCMF` закрывает базовый путь связи frontend-модулей с backend.

Его роль такая:

- дать модулю устойчивый способ читать и изменять backend-состояние;
- дать клиенту единый приведенный вид HTTP-ошибки;
- перенести `correlation-id`, `request-id` и `Retry-After` в клиентскую
  диагностику и поведение;
- связать результат HTTP-операции с состоянием модуля;
- при необходимости продолжить локальную причинно-следственную цепочку через
  `bus`.

В `LCMF` HTTP не является случайной внешней деталью вокруг модуля.

Но это не означает, что каждый модуль строит свою отдельную транспортную
подсистему.

Практическая картина такая:

- app-level слой создает общий HTTP-рантайм;
- `lcmf-http` дает общие транспортные средства;
- модуль владеет своими HTTP-сценариями;
- модуль сам решает, как результат операции отражается в его состоянии.

## 2. Где HTTP-рантайм живет в обычном приложении

HTTP-рантайм создается на app-level и передается `LCMF`-модулям через
зависимости.

Обычная картина такая:

- приложение создает один общий HTTP-рантайм;
- рантайм знает про базовый URL, общие headers, auth policy, разбор ответов и
  нормализацию ошибок;
- модули используют его для своих операций;
- модуль не должен каждый раз заново собирать транспортный слой без особой
  причины.

Небольшой пример:

Это app-level пример создания общего HTTP-рантайма.

```clojure
(ns app.system
  (:require [lcmf.http :as http]))

(defn app-transport [request]
  ...)

(defn make-system []
  (let [http-runtime (http/make-runtime {:base-url "/api"
                                         :transport app-transport})]
    {:http http-runtime}))
```

Смысл этого примера простой:

- HTTP-рантайм принадлежит приложению;
- транспортная политика остается общей;
- модуль получает уже готовую инфраструктурную зависимость;
- модуль владеет своими сценариями, а не низкоуровневой транспортной
  механикой как таковой.

## 3. Что `lcmf-http` должно давать

В первой очереди `LCMF` поддерживающая библиотека `lcmf-http` должна закрывать
минимально необходимую клиентскую основу.

Практически от нее ожидается следующее:

- создание общего рантайма;
- единый вход для выполнения запросов;
- приведение error response к единому виду;
- разбор `Retry-After`;
- единое место для транспортных пограничных случаев;
- по возможности, удобные механизмы для отмены и защиты от устаревших
  результатов.

Важно:

- `HTTP.md` не должен быть только справкой по `lcmf-http`;
- но и не должен делать вид, что поддерживающей библиотеки не существует;
- библиотека здесь вторична по отношению к frontend-модели, а не наоборот.

## 4. Базовая модель HTTP-ошибки

Frontend в `LCMF` должен работать с единым приведенным видом ошибки для
серверного error response.

Ожидаемая практическая форма:

```clojure
{:status 409
 :code "slot_not_open"
 :message "Slot not open"
 :details nil
 :correlation-id "c1a2..."
 :request-id "r9f0..."
 :retry-after-sec nil}
```

Эта форма продолжает backend HTTP-контракт и не должна ему противоречить.

Практические правила:

- для клиентской логики нужно ориентироваться на `code`, а не на `message`;
- `message` нужен как fallback для человека;
- `details` нужно обрабатывать как машинно-читаемую структуру;
- точные тексты ошибки нельзя считать стабильным контрактом;
- `retry-after-sec` имеет смысл прежде всего для `429` и `503`.

## 5. Что не равно обычной HTTP-ошибке

Для фронтенда важно не смешивать в одну кучу разные классы проблем.

Есть как минимум три разных случая:

- backend ответил HTTP-ответом с error contract;
- транспорт завершился клиентской неуспешностью без нормального HTTP-ответа;
- запрос был отменен или его результат уже устарел для текущего экрана.

Это не один и тот же сценарий.

Для малого и среднего проекта здесь особенно важно не расползтись в ad hoc
логику по модулям.

Поэтому `lcmf-http` или app-level слой должны дать проекту единое место для
таких случаев:

- `offline`;
- timeout;
- network failure;
- CORS/network rejection;
- abort/cancel;
- неожиданная форма ответа.

Этот документ не обязан сейчас фиксировать полный вид для каждого такого
случая.

Но документ должен явно признать:

- frontend живет не только телом серверной ошибки;
- транспортные сбои и отмена являются нормальной частью модели.

## 6. `correlation-id` и `request-id`

Для HTTP в `LCMF` важно различать два идентификатора:

- `correlation-id` — идентификатор общей цепочки;
- `request-id` — идентификатор конкретного HTTP-запроса.

Практический смысл:

- `correlation-id` помогает связать frontend-проблему с backend-трассой;
- `request-id` помогает найти конкретный запрос среди других;
- если доступны оба значения, сохранять нужно оба.

Для браузера это означает:

- JavaScript увидит эти headers только если приложение открыло их через
  `Access-Control-Expose-Headers`;
- практический минимум: `x-correlation-id, x-request-id`.

Важно не сделать из этого неверный вывод, будто frontend может спокойно
строить на наличии этих полей основную рабочую модель.

Более точная картина такая:

- эти поля очень полезны для отладки и поддержки;
- frontend должен уметь ими пользоваться;
- но основная бизнес-логика клиента не должна зависеть от того, удалось ли их
  получить в конкретном ответе.

## 7. Основные функции

Ниже описан ожидаемый облик поддерживающей библиотеки `lcmf-http`.

Это не отменяет архитектурную часть документа.
Наоборот, библиотечный API должен быть носителем именно этой frontend-модели.

### `make-runtime`

Создает HTTP-рантайм приложения.

```clojure
(require '[lcmf.http :as http])

(def app-http
  (http/make-runtime {:base-url "/api"
                      :transport app-transport}))
```

`lcmf-http` требует явно указать `:transport`. Этот transport adapter может быть поверх:

- `fetch`;
- тонкой оберткой над выбранной библиотекой клиента;
- другого транспортного API, если проекту это реально нужно.

Важно:

- внешний API `LCMF` не должен зависеть от конкретного транспортного API;
- модуль должен работать с `request!` и приведенной ошибкой, а не с сырым
  поведением `fetch` в каждом месте отдельно.

### `request!`

Выполняет HTTP-запрос.

Сигнатура:

```clojure
(request! runtime request-map)
```

Где `request-map` обычно включает:

- `:method`
- `:path`
- `:query`
- `:headers`
- `:json` или `:body`

Пример:

Это пример использования HTTP-рантайма из кода `LCMF`-модуля.

```clojure
(http/request! app-http
               {:method :get
                :path "/bookings"
                :query {:user-id "u-alice"}})
```

### `normalize-error`

Приводит HTTP error response к единому клиентскому виду.

Пример:

Это пример уровня библиотеки `lcmf.http`, а не пример кода отдельного модуля.

```clojure
(defn normalize-error [response body]
  {:status (:status response)
   :code (or (:code body) "unknown_error")
   :message (or (:message body) "Request failed")
   :details (:details body)
   :correlation-id (or (get body :correlation-id)
                       (get-in response [:headers "x-correlation-id"]))
   :request-id (or (get body :request-id)
                   (get-in response [:headers "x-request-id"]))
   :retry-after-sec (parse-retry-after (:headers response))})
```

### `parse-retry-after`

Разбирает `Retry-After`, если он есть.

```clojure
(defn parse-retry-after [headers]
  (let [raw (get headers "retry-after")
        n (js/Number raw)]
    (when (and (js/Number.isFinite n) (pos? n))
      n)))
```

Это пример уровня библиотеки `lcmf.http`.

### `compute-delay-ms`

Вычисляет задержку перед следующим повтором.

```clojure
(defn compute-delay-ms [attempt retry-after-sec]
  (if retry-after-sec
    (* retry-after-sec 1000)
    (let [base (min (* 1000 (Math/pow 2 attempt)) 15000)
          jitter (rand-int 250)]
      (+ base jitter))))
```

Это пример уровня библиотеки `lcmf.http`.

## 8. Как читать основные статусы

Для `LCMF` базовая практическая картина такая:

- `400` и `422` — ошибки входных данных, их не нужно автоматически повторять;
- `401` — сигнал потери аутентификации или сессии;
- `403` — отказ в доступе;
- `404` — отсутствие ресурса или предметный результат, а не обязательно авария;
- `409` — конфликт с текущим состоянием;
- `429` — ограничение частоты или временная блокировка;
- `503` — временная деградация или недоступность.

Важно:

- `401`, `403`, `429`, `503` являются штатными ответами;
- клиентское поведение должно ветвиться по `status` и `code`, а не по тексту;
- `429` и `503` требуют паузы перед повтором.

## 9. Повторы запросов и `Retry-After`

Если пришел `429` или `503`:

- сначала нужно проверить `Retry-After`;
- если он есть, использовать его как базовую задержку;
- если его нет, использовать fallback backoff;
- число автоматических повторов должно быть ограничено.

Но этого правила недостаточно самого по себе.

Для фронтенда важно еще различать:

- идемпотентное чтение;
- пользовательская операция записи;
- запрос переднего плана, где пользователь прямо ждет результата;
- фоновое обновление, где важнее мягкое восстановление состояния.

Поэтому практическое правило такое:

- автоматический повтор не должен становиться общей реакцией на статус;
- safest default для автоповтора — ограниченные случаи идемпотентного чтения;
- пользовательские write-операции требуют большей осторожности;
- повтор не должен превращаться в бесконечный цикл или в скрытый повтор
  пользовательского действия.

Пример:

Это пример вспомогательной функции уровня `LCMF`-модуля или app-level
клиентского рантайма.

```clojure
(defn retry-request! [request! attempt error]
  (let [delay-ms (compute-delay-ms attempt (:retry-after-sec error))]
    (js/setTimeout
     (fn []
       (request!))
     delay-ms)))
```

## 10. Связь HTTP и состояния модуля

HTTP-операция не завершает свою роль на уровне транспорта.

`LCMF`-модуль должен сам решить:

- как операция отражается в его состоянии;
- какие служебные поля состояния меняются;
- публикуется ли по итогу сообщение в `bus`.

### Пример связи HTTP и состояния

Это пример операции уровня `LCMF`-модуля.

```clojure
(defn load-bookings!
  [{:keys [state http]}]
  (swap! state assoc :load-request {:status :pending
                                    :error nil})
  (-> (http/request! http {:method :get :path "/bookings"})
      (.then (fn [bookings]
               (swap! state assoc
                      :items (into {} (map (juxt :id identity)) bookings)
                      :load-request {:status :ok
                                     :error nil})
               bookings))
      (.catch (fn [error]
                (swap! state assoc
                       :load-request {:status :error
                                      :error error})
                (throw error)))))
```

Здесь важен именно ownership:

- состояние обновляет сам модуль-владелец;
- общая транспортная механика остается общей;
- смысл результата принадлежит модулю.

## 11. Отмена, гонки и устаревшие ответы

Для frontend-модели этого документа важно явно признать:

- не каждый завершившийся запрос должен менять текущее состояние;
- запрос может быть отменен;
- старый ответ может прийти позже нового;
- экран или сценарий могут уже перестать считать результат актуальным.

Это не edge-case, а обычная жизнь UI.

Поэтому `lcmf-http` или app-level слой должны давать проекту способ не
размазывать такие решения по всем модулям.

На этом уровне важно хотя бы удержать правило:

- отмена запроса и обработка устаревших ответов являются нормальной частью
  frontend-модели HTTP.

Точный API для этого может уточняться отдельно.

Если проекту нужен типовой stale/revalidate или latest-only helper, такой слой
лучше строить отдельно поверх `lcmf-http`, а не вшивать в сам `lcmf-http`.

## 12. Связь HTTP и шины

HTTP и `bus` не конкурируют друг с другом.

Обычная картина в `LCMF` такая:

- модуль выполняет HTTP-операцию;
- модуль отражает результат в своем состоянии;
- модуль при необходимости публикует сообщение в `bus`;
- другие модули реагируют на сообщение, не зная о конкретной HTTP-операции.

Важно удержать слово "при необходимости".

Это означает:

- не всякий успешный HTTP-запрос должен продолжаться через `bus`;
- `bus` нужен там, где действительно возник межмодульный факт;
- если результат важен только самому модулю, достаточно локального обновления
  состояния.

### Пример связи HTTP и шины

Это пример операции уровня `LCMF`-модуля.

```clojure
(defn create-booking!
  [{:keys [state http bus]} payload]
  (-> (http/request! http
                     {:method :post
                      :path "/bookings"
                      :json payload})
      (.then (fn [booking]
               (swap! state assoc-in [:items (:id booking)] booking)
               (bus/publish! bus
                             :booking/created
                             {:id (:id booking)
                              :user-id (:user-id booking)
                              :slot-id (:slot-id booking)}
                             {:module :booking})
               booking))))
```

## 13. Запросы переднего и фонового плана

Для малого и среднего проекта полезно различать два режима:

- запрос переднего плана;
- фоновый запрос обновления или синхронизации.

Это различие важно потому, что у них часто разный UX:

- разная форма состояния загрузки;
- разная форма состояния ошибки;
- разное отношение к автоматическому повтору;
- разная реакция на временную деградацию.

Примеры:

- отправка формы в переднем плане не должна тихо растворяться в фоне;
- фоновое обновление может сохранить старые данные и попробовать позже
  восстановить согласованность.

Этот документ не обязан навязывать один UI-паттерн.

Но frontend-спецификация должна хотя бы назвать это различие как нормальное.

## 14. Что важно не делать

В `LCMF` важно не делать следующего:

- не строить клиентскую логику на точных текстах `message`;
- не разбирать HTTP-ошибки по-разному в каждом модуле;
- не терять `correlation-id` и `request-id`, если они доступны;
- не делать websocket заменой начального HTTP-чтения;
- не повторять `429` и `503` без паузы;
- не превращать повтор в скрытый дубль любого пользовательского действия;
- не смешивать HTTP error response, transport failure и cancel как один и тот
  же случай;
- не передавать секреты в URL и query string.

## 15. Минимальный итог

HTTP в `LCMF` — это базовый путь связи frontend-модуля с backend и естественная
точка входа для поддерживающей библиотеки `lcmf-http`.

App-level слой создает общий HTTP-рантайм.
Модуль использует его для своих операций, сам отражает результат в своем
состоянии и при необходимости продолжает цепочку через `bus`.

При этом frontend-модель HTTP должна учитывать не только error contract
backend-а, но и обычную реальность клиента:

- транспортные сбои;
- отмену запросов;
- устаревшие ответы;
- осторожный автоматический повтор;
- различие между запросами переднего и фонового плана.
