# Документация: WebSocket во frontend

Этот документ описывает роль websocket в `LCMF`.

Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для app-level слоя, который владеет транспортной инфраструктурой, см.
[`APP.md`](./APP.md).
Для шины, через которую модули координируются между собой, см.
[`BUS.md`](./BUS.md).

Важно держать этот текст в согласии с backend-ориентирами:

- [`WEBSOCKET.md`](https://github.com/algebrain/lcmm-docs/blob/main/docs/WEBSOCKET.md)
- [`WS_APP_LEVEL.md`](https://github.com/algebrain/lcmm-docs/blob/main/docs/WS_APP_LEVEL.md)
- [`WS_MODULE_LEVEL.md`](https://github.com/algebrain/lcmm-docs/blob/main/docs/WS_MODULE_LEVEL.md)

Этот документ не придумывает другую websocket-модель.
Он описывает, как эта модель проживается на стороне фронтенда.

## 1. Главный принцип

В `LCMF` websocket является transport уровня приложения.

Это означает:

- websocket endpoint принадлежит app-level слою;
- `LCMF`-модуль не владеет websocket transport;
- `LCMF`-модуль публикует и потребляет сообщения через `bus`;
- приложение принимает внешние websocket-сообщения и доставляет push в
  браузер.

Именно в таком порядке это и нужно понимать.
WebSocket не становится прямым API отдельного модуля.

## 2. Когда websocket нужен

WebSocket нужен тогда, когда приложению действительно нужен push в браузер для
конкретного realtime-сценария.

Практический смысл здесь узкий:

- websocket не нужен "просто потому, что в системе есть `bus`";
- websocket не является вторым главным каналом архитектуры;
- websocket включается там, где у приложения есть реальная польза от живой
  доставки обновлений.

Типичные случаи:

- живое обновление статуса;
- доставка уведомлений;
- совместная работа с одним объектом;
- другие сценарии, где задержка обычного перечитывания по HTTP уже слишком
  заметна.

Базовый рабочий сценарий такой:

- начальное состояние загружается по HTTP;
- websocket доставляет последующие обновления;
- после разрыва соединения клиент при необходимости восстанавливает точную
  картину через HTTP.

Это означает:

- websocket не заменяет HTTP;
- экран не должен зависеть только от постоянного сокет-соединения;
- потеря соединения не должна делать приложение логически неработоспособным.

## 3. Что принадлежит приложению

App-level слой владеет:

- websocket endpoint;
- правилами открытия соединения;
- проверкой `Origin`;
- аутентификацией и транспортной авторизацией;
- хранением соединений и подписок;
- внешней формой сообщений для браузера;
- маршрутизацией по темам;
- доставкой сообщений подписчикам;
- app-owned ingress event для входящих сообщений.

### Пример app-level создания websocket runtime

Это app-level пример.

```clojure
(defn make-system []
  (let [ws-runtime (ws/make-runtime)]
    {:ws ws-runtime}))
```

## 4. Чем владеет `LCMF`-модуль

`LCMF`-модуль владеет:

- своими предметными сообщениями;
- своей логикой состояния;
- своей доменной интерпретацией входящих фактов;
- своими HTTP-контрактами;
- своими bus-контрактами;
- своими read-контрактами.

`LCMF`-модуль не должен владеть:

- websocket endpoint;
- внешней topic model для браузера;
- внешней формой websocket-сообщения;
- raw websocket input contract.

## 5. Роль `lcmf-ws`

`lcmf-ws` нужно понимать как поддерживающую библиотеку app-level websocket
слоя.

Ее роль в том, чтобы снять повторяющийся транспортный и сессионный код там, где
это действительно полезно:

- создание websocket runtime;
- работа с соединениями и подписками;
- доставка сообщений;
- базовые служебные операции;
- при необходимости удобные средства для обработки жизненного цикла соединения.

Ее роль не в том, чтобы забрать у приложения архитектурные решения.

Поэтому `lcmf-ws` не должна незаметно брать на себя то, что остается решением
app-level слоя:

- нужен ли websocket в этом проекте вообще;
- как устроен внешний контракт сообщений;
- как устроены правила доступа;
- как связаны reconnect и resync;
- как transport-события отражаются в клиентском состоянии.

## 6. Исходящий поток в браузер

Обычная модель исходящего потока такая:

- `LCMF`-модуль публикует предметное сообщение в `bus`;
- app-level websocket слой подписывается на это сообщение;
- приложение решает, нужно ли доставлять его в браузер;
- приложение строит сообщение для браузера;
- приложение отправляет его через websocket transport.

### Пример сообщения, которое публикует `LCMF`-модуль

Это пример уровня `LCMF`-модуля.

```clojure
(bus/publish! bus
              :booking/created
              {:id booking-id
               :slot-id slot-id
               :user-id user-id}
              {:module :booking})
```

### Пример app-level проекции сообщения в browser-facing websocket message

Это app-level пример.

```clojure
(bus/subscribe! bus
                :booking/created
                (fn [_ envelope]
                  (let [user-id (get-in envelope [:payload :user-id])
                        message {:type "event"
                                 :event "booking/created"
                                 :topic ["user" user-id]
                                 :payload {:bookingId (get-in envelope [:payload :id])
                                           :userId user-id
                                           :slotId (get-in envelope [:payload :slot-id])}
                                 :correlationId (:correlation-id envelope)}]
                    (ws/send-to-topic! ws-runtime
                                       [:user user-id]
                                       message))))
```

Во втором примере именно приложение решает:

- нужно ли вообще это событие браузеру;
- кому его доставлять;
- что именно уйдет наружу;
- в каком виде это будет выглядеть.

Важно не сделать отсюда лишний вывод.

Это не означает, что websocket является естественным продолжением любого
предметного события.

Правильная мысль здесь такая:

- websocket-путь появляется только там, где у приложения есть конкретный
  realtime-сценарий;
- наличие `bus` само по себе не означает, что событие должно уйти в браузер;
- решение о websocket-доставке остается практическим app-level решением.

## 7. Входящий поток из браузера

Если websocket используется не только для push, но и для входящих сообщений,
приложение не должно публиковать доменные сообщения от имени модуля.

Обычная модель входящего потока такая:

- браузер присылает websocket message;
- приложение проверяет транспортные правила;
- приложение нормализует message;
- приложение публикует app-owned ingress event в `bus`;
- `LCMF`-модуль при необходимости подписывается на это ingress event и сам
  решает, какова его предметная интерпретация.

Зафиксированное ingress event:

- `:app/external-message-received`

### Пример app-owned ingress payload

Это пример app-level контракта, который может читать `LCMF`-модуль.

```clojure
{:message-kind :command
 :message-name "booking/create"
 :payload {:slot-id "slot-09-00"}
 :subject {:user-id "u-alice"}
 :source {:channel :websocket
          :transport :ws
          :session-id "..."
          :remote-addr "..."
          :origin "http://localhost:3006"}
 :correlation-id "..."
 :received-at 1710000000}
```

Важно:

- это app-owned contract;
- это не raw websocket frame;
- это не доменное сообщение `LCMF`-модуля.

Эту форму полезно понимать как опорную, а не как единственно возможную
реализацию каждого frontend runtime.

Главное здесь не точный состав полей, а инварианты:

- transport-вход сначала нормализуется на app-level;
- модуль не получает raw websocket payload;
- доменная интерпретация остается в модуле.

### Пример реакции `LCMF`-модуля на ingress event

Это пример уровня `LCMF`-модуля.

```clojure
(bus/subscribe! bus
                :app/external-message-received
                (fn [_ envelope]
                  (let [{:keys [message-name payload]} (:payload envelope)]
                    (when (= message-name "booking/create")
                      (create-booking! {:slot-id (:slot-id payload)}))))
                {:module :booking})
```

## 8. Минимальный объем первой фазы

Первая фаза websocket в `LCMF` включает:

- `subscribe`;
- `unsubscribe`;
- `ping`;
- server push от приложения к браузеру.

Первая фаза не обязана включать:

- произвольный command API по websocket как основной путь системы;
- гарантированную доставку после разрыва связи;
- offline queue;
- распределенное хранение соединений.

Это нужно понимать не как запрет на развитие, а как сознательно узкий стартовый
объем.

Смысл здесь такой:

- сначала закрыть понятный и обозримый websocket-сценарий;
- не превращать первую реализацию в тяжелую платформу "на все случаи";
- расширять контур только по реальной потребности проекта.

## 9. Внешний контракт сообщений

Внутреннее предметное сообщение и browser-facing websocket message не обязаны
совпадать один к одному.

Приложение может:

- убрать лишние поля;
- переименовать поля в удобную внешнюю форму;
- добавить технические поля вроде `correlationId`.

Приложение не должно:

- менять предметный смысл события;
- придумывать новый предметный статус, которого модуль не объявлял.

### Пример browser-facing сообщений первой фазы

Это пример app-level websocket-контракта.

```json
{"type":"subscribed","topic":["user","u-alice"]}
{"type":"unsubscribed","topic":["user","u-alice"]}
{"type":"pong"}
{"type":"error","code":"subscription_rejected","message":"Subscription rejected"}
{"type":"event","event":"booking/created","topic":["user","u-alice"],"payload":{"bookingId":"...","userId":"u-alice","slotId":"..."},"correlationId":"..."}
```

Этот пример не должен читаться как обязательный словарь любой реализации.

Он фиксирует минимальный практический baseline:

- у клиента есть понятные служебные сообщения;
- у предметного push есть внешняя форма;
- внешний контракт определяется приложением, а не модулем.

## 10. Reconnect и resync

Reconnect и восстановление точной картины состояния — это не одно и то же.

После reconnect у клиента снова появляется websocket-канал.
Но этого самого по себе недостаточно, чтобы считать экран полностью
согласованным.

Поэтому практическая модель в `LCMF` такая:

- reconnect восстанавливает соединение;
- подписки при необходимости создаются заново;
- если экрану нужна точная картина, состояние восстанавливается по HTTP;
- только после этого экран можно считать снова согласованным.

### Два допустимых режима реализации

Здесь полезно явно различать два случая.

#### 1. Выбранный websocket runtime уже закрывает часть жизненного цикла

Например, библиотека уже берет на себя:

- reconnect;
- повторное открытие соединения;
- повторную подписку;
- часть транспортных повторов;
- часть управления состоянием соединения.

Тогда проект не обязан дублировать эту логику вручную.

Но архитектурные инварианты остаются прежними:

- reconnect сам по себе не равен полностью восстановленной согласованности;
- HTTP остается каналом восстановления точной картины;
- модуль не становится websocket-aware;
- app-level слой продолжает владеть внешним и ingress-контрактом.

#### 2. Выбранный runtime этого не закрывает

Тогда приложение должно организовать это само.

В частности, проекту придется явно решить:

- как делать reconnect;
- как повторно поднимать подписки;
- когда нужен resync через HTTP;
- как экран переживает деградацию соединения.

### Пример app-level поведения после закрытия соединения

Это app-level пример.

```clojure
(defn on-close! [{:keys [http reconnect! reload-screen-state!]}]
  (reload-screen-state! http)
  (reconnect!))
```

Этот пример полезно читать как один из возможных практических вариантов, а не
как единственно допустимую форму реализации.

## 11. Transport-событие и состояние экрана

Для frontend-модели важно удержать еще одну границу:

- transport-событие не обязано буквально равняться пользовательскому
  последствию;
- получение push не обязано автоматически менять текущий экран;
- экран или app-level client runtime сам решает, как входящее сообщение
  отражается в клиентском состоянии.

Это особенно важно в таких случаях:

- пришло событие для уже неактуального экрана;
- локальное состояние экрана уже ушло вперед;
- есть optimistic update;
- соединение деградирует, но UI не должен распадаться на бинарное
  "все работает / все сломалось".

Этот документ не должен навязывать одну UI-архитектуру.
Но он должен честно признать, что transport receipt и state update — это не
одно и то же.

## 12. Безопасность

Минимальный baseline такой:

- auth на handshake уровне приложения;
- проверка `Origin`;
- отсутствие секретов в URL и query string;
- authorization на каждый `subscribe`;
- единый безопасный отказ без лишних деталей;
- reconnect проходит ту же проверку заново;
- raw payload не логируется по умолчанию.

## 13. Что важно не делать

В `LCMF` важно не делать следующего:

- не давать `LCMF`-модулю прямую зависимость от websocket transport;
- не публиковать доменные сообщения модуля от имени `app`;
- не превращать raw websocket payload в контракт модуля;
- не подменять HTTP начальной загрузки одним websocket;
- не делать вид, что websocket нужен просто потому, что в системе есть `bus`;
- не смешивать транспортные решения и доменную интерпретацию.

## 14. Минимальный итог

WebSocket в `LCMF` — это app-level transport, который дополняет HTTP и нужен
для узких realtime-сценариев, где он действительно оправдан.

`LCMF`-модуль не владеет websocket endpoint и не должен знать transport-детали.
Он публикует и потребляет сообщения через `bus`, а приложение само решает:

- нужен ли websocket для данного сценария;
- что именно доставлять в браузер;
- как устроен внешний контракт;
- как устроены reconnect и resync;
- как transport-события отражаются в клиентском состоянии.

`lcmf-ws` при этом должна оставаться поддерживающей библиотекой app-level
слоя, а не новым центром архитектуры.
