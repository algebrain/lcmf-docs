# Документация: Контракт frontend-модуля

Этот документ описывает, как в `LCMF` задается контракт frontend-модуля.
Задача контракта в том, чтобы зафиксировать:

- какие внешние поверхности модуль объявляет наружу;
- какие внешние поверхности модуль ожидает от соседних модулей и app-level
  слоя;
- как такой контракт читать без доступа к внутреннему устройству модуля;
- как сделать модуль понятной библиотекой для соседних модулей.

Для общей архитектурной рамки см. [`ARCH.md`](./ARCH.md).
Для формы отдельного модуля см. [`MODULE.md`](./MODULE.md).
Для `bus` см. [`BUS.md`](./BUS.md).
Для публичных синхронных чтений см. [`REGISTRY.md`](./REGISTRY.md).
Для HTTP и realtime поверхностей см. [`HTTP.md`](./HTTP.md) и
[`WEBSOCKET.md`](./WEBSOCKET.md).

## 1. Назначение

Контракт `LCMF`-модуля нужен в первую очередь для разработчиков соседних
модулей.

Ключевой сценарий такой:

- модуль разрабатывается отдельно;
- внешний разработчик не должен читать его внутренний state и внутренние
  функции;
- внешний разработчик должен понимать, какие сигналы модуль публикует, какие
  чтения предоставляет и какие внешние зависимости ожидает.

Контракт не описывает внутреннюю организацию модуля.
Он описывает его публичную поверхность.

## 2. Что решает контракт

Контракт frontend-модуля в `LCMF` решает такие задачи:

- отделяет публичную поверхность модуля от его внутреннего состояния;
- позволяет соседнему модулю понимать, как с этим модулем взаимодействовать;
- делает зависимости модуля явными и проверяемыми;
- не дает межмодульному взаимодействию скатиться в чтение чужих внутренностей.

Если у модуля нет явного контракта, соседний код начинает зависеть от:

- внутренней формы state;
- случайных имен внутренних функций;
- незафиксированных shape сообщений;
- догадок о том, какие данные можно читать и какие нельзя.

Для `LCMF` это считается плохой практикой.

## 3. Что контрактом не является

Контрактом frontend-модуля не являются:

- точная внутренняя форма участка состояния;
- внутренние helper-функции;
- локальная файловая структура модуля;
- случайные временные поля request lifecycle;
- технические детали конкретной реализации store, atom или component tree.

Также важно:

- `LCMF` не описывает здесь view-layer как таковой;
- этот документ не задает правила построения экранов;
- этот документ не превращает raw websocket transport в модульный API.

Контракт описывает только те поверхности, на которые внешний код вправе
опираться.

## 4. Основные контрактные поверхности модуля

Для `LCMF` practically useful картина такая: у frontend-модуля есть несколько
нормальных межмодульных поверхностей.

### 4.1 Публикуемые сообщения

Модуль может публиковать сообщения в `bus`.

Это одна из главных внешних поверхностей модуля, потому что именно через нее в
`LCMF` строится основная межмодульная координация.

Если модуль публикует сообщение наружу, внешний код должен понимать:

- имя сообщения;
- краткий смысл сообщения;
- shape payload;
- кто считается владельцем этого сообщения.

### 4.2 Потребляемые сообщения

Если модуль зависит от сообщений другого модуля или app-level слоя, это тоже
должно быть видно из контракта.

Внешнему разработчику важно понимать:

- какие входящие сообщения модуль ожидает;
- какие из них являются обязательными;
- какая минимальная форма payload ему нужна;
- является ли это модульной или app-level зависимостью.

### 4.3 Публичные синхронные чтения

Если модуль экспортирует публичные чтения через `registry`, это является
полноценной контрактной поверхностью.

Внешний код должен понимать:

- `provider-id`;
- какие аргументы принимает provider;
- что именно он возвращает;
- какие данные считаются публичными, а какие остаются внутренними.

### 4.4 Обязательные внешние чтения

Если модуль сам зависит от чужих providers, это тоже должно быть явно
зафиксировано.

Такой контракт нужен, чтобы:

- соседний модуль понимал, какие providers он обязан предоставить;
- app-level сборка могла проверить зависимости до старта;
- внешний код не угадывал обязательные чтения по внутреннему коду модуля.

### 4.5 Публичные функции верхнего уровня

Если модуль экспортирует верхнеуровневые функции как часть своей публичной
поверхности, это тоже входит в контракт.

Это может быть, например:

- `init!`;
- функция загрузки данных;
- функция запуска пользовательского действия;
- функция получения публичного дескриптора модуля.

Но в контракт должны попадать только реально публичные функции.
Внутренние helper-функции сюда не входят.

### 4.6 Сетевые поверхности

Хотя `LCMF` не является transport-first архитектурой, у модуля может быть
важная сетевая поверхность, которую полезно видеть соседнему модулю.

Речь идет о таких вещах:

- какие модульные операции инициируют HTTP-взаимодействие;
- с какими backend endpoint-ами это связано;
- какие нормализованные результаты и ошибки появляются по итогу;
- какие сообщения публикуются как следствие сетевой операции.

Это нужно не для полного описания transport runtime, а для понимания
межмодульного поведения.

### 4.7 App-owned зависимости

Некоторые зависимости модуля принадлежат не соседнему модулю, а приложению.

Например:

- app-owned ingress event;
- app-level auth state;
- app-level realtime notification contract;
- app-level HTTP runtime assumptions.

Если модуль существенно зависит от такой поверхности, это полезно отражать в
контракте отдельно, чтобы не смешивать ее с соседним модулем-владельцем.

## 5. Каноническая структура контракта

В `LCMF` у модуля может быть до пяти разных внешних поверхностей:

1. `exports`
2. `events`
3. `read_providers`
4. `http`
5. `app_contracts`

Не каждый модуль обязан использовать все из них.
Но если поверхность реально существует, ее полезно описывать явно.

Practically useful shape:

```yaml
contract_version: "1.0"
module:
  name: "booking"
  owner: "team-booking"
  summary: "Управление локальным состоянием и действиями бронирования"

exports:
  functions: []

events:
  publishes: []
  subscribes: []

read_providers:
  provides: []
  requires: []

http:
  operations: []

app_contracts:
  consumes: []

types:
  schemas: {}

narrative:
  purpose: ""
  scenarios: []
  dependencies: []

compatibility:
  module_contract_version: "0.1.0"
  backwards_compatible_with: []
```

Правило компактности:

- пустые коллекции допустимо опускать;
- но реально существующие публичные поверхности лучше не прятать.

## 6. Контракт публикуемых сообщений

Если модуль публикует сообщения в `bus`, для каждого такого сообщения полезно
фиксировать:

- `event`;
- `summary`;
- `payload`;
- `owner_module`.

Минимальный пример:

```yaml
events:
  publishes:
    - event: "booking/created"
      summary: "Модуль создал локально подтвержденную бронь"
      owner_module: "booking"
      payload:
        type: object
        required: [id, slot_id, user_id]
```

Что здесь важно:

- имя события должно быть стабильным;
- payload должен описывать публичный shape, а не внутренний state целиком;
- владелец события должен быть ясен внешнему коду.

## 7. Контракт потребляемых сообщений

Если модуль подписывается на внешние сообщения, это нужно описывать отдельно.

Для каждого входящего сообщения полезно фиксировать:

- `event`;
- `summary`;
- `source`;
- `payload`;
- `required`.

Минимальный пример:

```yaml
events:
  subscribes:
    - event: "accounts/signed-out"
      summary: "Модуль очищает данные пользователя после выхода"
      source: "module"
      required: false
      payload:
        type: object
        required: [user_id]
```

Поле `source` practically useful различать хотя бы так:

- `module`
- `app`

Это помогает понять, чья это контрактная поверхность.

## 8. Контракт публичных чтений

Если модуль что-то отдает через `registry`, это должно быть читаемо без знания
его внутренней формы state.

Для каждого provider полезно фиксировать:

- `provider_id`;
- `summary`;
- `args`;
- `returns`.

Минимальный пример:

```yaml
read_providers:
  provides:
    - provider_id: "booking/get-by-id"
      summary: "Вернуть публичную форму брони по id"
      args:
        type: object
        required: [booking_id]
      returns:
        type: object
```

Главное правило:

- provider описывает внешний read-case;
- provider не должен обещать весь внутренний slice состояния.

## 9. Контракт требуемых чтений

Если модулю нужны внешние providers, это должно быть видно из контракта.

Для каждого required provider полезно фиксировать:

- `provider_id`;
- `summary`;
- `required`.

Минимальный пример:

```yaml
read_providers:
  requires:
    - provider_id: "catalog/get-slot-by-id"
      summary: "Нужен для проверки доступности слота"
      required: true
```

Это особенно важно для app-level fail-fast проверки и для независимой
разработки модулей.

## 10. HTTP как вторичная контрактная поверхность

Для `LCMF` HTTP важен, но в контракте frontend-модуля он не должен становиться
главной осью документа.

HTTP-поверхность полезно отражать тогда, когда внешний код должен понимать:

- какие модульные операции инициируют сетевое действие;
- какой backend endpoint с этим связан;
- какой shape успеха и нормализованной ошибки ожидается;
- какие публичные сообщения или публичные данные обновляются по итогу.

Practically useful пример:

```yaml
http:
  operations:
    - operation_id: "create_booking"
      summary: "Создать бронь через backend API"
      request:
        type: object
        required: [slot_id, user_id]
      success:
        type: object
      error:
        type: object
      emits: ["booking/created", "booking/create-failed"]
```

Важно:

- это не замена `HTTP.md`;
- это описание модульной поверхности, а не полного transport runtime.

## 11. App-owned контракты

Некоторые важные входы модуля принадлежат приложению, а не соседнему модулю.

Это может быть:

- `:app/external-message-received`;
- app-owned auth/session contract;
- app-owned browser-facing realtime contract;
- app-owned routing или lifecycle signal, если он действительно входит в
  публичную интеграцию модуля.

Для таких поверхностей полезно иметь отдельный блок, чтобы не смешивать их:

- с модульными событиями;
- с чужими providers;
- с внутренними деталями app wiring.

## 12. Narrative-блок обязателен

Одних схем и имен недостаточно.

В `LCMF` narrative-блок особенно важен, потому что внешний разработчик должен
понимать не только shape данных, но и смысл модульной поверхности.

`narrative` полезно строить вокруг трех вещей:

- `purpose` — зачем модуль существует;
- `scenarios` — какие ключевые сценарии он покрывает;
- `dependencies` — какие соседние модули и app-level контракты для него важны.

Если контракт можно прочитать только как набор разрозненных операций и
сообщений, он еще недостаточно полезен для соседнего разработчика.

## 13. Совместимость и breaking changes

Контракт frontend-модуля тоже должен версионироваться.

Breaking changes здесь вполне реальны.

Примеры breaking change:

- удалили публикуемое событие;
- переименовали событие;
- изменили shape payload у уже существующего события;
- удалили `provider_id`;
- изменили аргументы публичной функции несовместимым образом;
- перестали возвращать поле, на которое внешний соседний модуль мог опираться.

Поэтому practically useful минимум такой:

- у контракта есть своя версия;
- breaking changes считаются отдельно от внутренних рефакторингов;
- внутренняя перестройка модуля не считается breaking, если публичная
  поверхность осталась прежней.

## 14. Минимальный итог

Контракт `LCMF`-модуля нужен для того, чтобы внешний код видел модуль как
понятную библиотеку, а не как случайный кусок state и внутренних функций.

Для `LCMF` наиболее важные поверхности контракта такие:

- публикуемые и потребляемые сообщения;
- публичные чтения и обязательные внешние providers;
- публичные функции верхнего уровня;
- сетевые операции как вторичная поверхность;
- app-owned зависимости;
- HTTP как дополнительная, а не главная поверхность.
